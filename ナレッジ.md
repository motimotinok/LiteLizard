## 2026-04-18: 個人開発では dev 常駐を基本にし必要時だけ worktree へ分離する運用が軽い

**カテゴリ**: フロー改善

**気づき**:
個人開発で常に branch や PR を前提にすると、レビュー、同期、マージの摩擦が大きくなりやすい。一方で、すべてを `dev` 直作業にすると、長引く変更や危険な変更を分離しづらい。普段は `dev` ブランチ常駐で進め、必要になった時だけ `worktree` を作って隔離し、レビュー後にローカルで `dev` へマージする形にすると、軽さと安全性のバランスが取りやすい。

**改善策・対応**:
通常の小さい修正は `dev` で進め、中規模以上の変更、長引く変更、並列したい変更だけ `worktree` を切る。取り込みは PR 必須にせず、ローカルレビュー後に `dev` へマージし、そのタイミングで必要な管理ファイルも更新する運用を基本候補にする。

## 2026-04-18: 新規開発テーマは面白さだけでなく今の自分に対して少し背伸び程度の難易度かも見る

**カテゴリ**: フロー改善

**気づき**:
LiteLizard のようなエディター系アプリは、見た目以上に実装すべき領域が広く、UI、状態管理、保存、構造編集、テストなど多方面の難しさが一気に乗ってくる。そのため、まだ習熟途中の段階でいきなり取り組むと、面白さはあっても実装コストが高すぎて詰まりやすい。新しく開発を始めるときは、「作っていて面白いか」に加えて、「今の自分で十分作れそうか」「少し難しいが手が届く範囲か」を事前に見た方が、継続しやすく学習効率も良さそう。

**改善策・対応**:
今後のテーマ選定では、アイデアの魅力だけで決めず、必要な実装範囲と現在の自分の技術レベルの差も軽く見積もる。難しすぎる題材は避けるか、より小さいスコープに分解して着手し、少し背伸びするくらいの難易度に収める。

## 2026-04-18: AGENTS.md には変動しやすい開発フローを書きすぎず安定情報だけ置く方が誤解を減らせる

**カテゴリ**: フロー改善

**気づき**:
LLM が最初に読むプロンプトファイルに具体的な開発フローや運用手順を書きすぎると、別 worktree や別セッションで動いているエージェントがその文脈を固定ルールだと誤解しやすい。また、ワークフローを試行錯誤している段階では、その変更のたびに `AGENTS.md` 側も追随修正が必要になり、メンテナンス負荷が高い。`AGENTS.md` には、重要ファイルの場所、プロジェクト全体でほぼ変わらない共通認識、前提ルールのような安定情報だけを置き、変わりやすい手順や運用フローは切り離した方が扱いやすい。特に並列開発では、LLM 自身にフロー全体を強く持たせるより、人間が統合する構成やフック・連鎖実行の仕組み側でワークフローを担保した方が、改善ループを回しやすそう。

**改善策・対応**:
`AGENTS.md` には不変に近い前提と重要ファイル案内だけを書く。変更されやすいワークフローは必要に応じてスキルや外部の仕組みに切り出し、プロンプト本体にはできるだけ埋め込まない。並列開発では、各 LLM に全体フロー理解を期待しすぎず、人間の統合やフック、チェーン式のタスク連携で補う設計を優先する。

## 2026-04-18: 実装の並列開発は同一セッションのサブエージェント分担より複数セッション運用の方が現状は扱いやすい

**カテゴリ**: フロー改善

**気づき**:
同一セッション内で複数のサブエージェントに並列実装させ、最後にマージする運用は、理屈の上では効率化できそうでも、実際にはセッション管理、レビュー状況、テスト状況、最終的な統合作業の把握が重くなりやすい。特に「今どの実装がどこまで進んだか」「どの検証が終わっているか」を追う負荷が高く、現状の仕組みやモデル性能だと安定運用しづらい。サブエージェントは探索や補助的な作業には有効でも、具体的な実装作業そのものを大量に並列で進める用途には、まだあまり向いていない感触がある。

**改善策・対応**:
実装の並列開発をしたい場合は、同一セッション内でサブエージェントを大量に束ねるより、素直に複数セッションを分けて進める方を優先する。サブエージェントは調査、要約、限定的な補助タスクに寄せ、実装本体の進行管理は独立したセッション単位で扱う方が状況把握しやすい。

## 2026-04-18: タスク管理は issue、方向性共有は別ダッシュボードに分ける方が扱いやすい可能性がある

**カテゴリ**: フロー改善

**気づき**:
WBS をタスク管理表として使い続けるより、個々のタスクは issue で管理し、全体の方向性・コンセプト・優先判断のための情報は別のダッシュボード的な概念に切り出した方が運用しやすい可能性がある。WBS は「全体像」と「タスク台帳」を同時に背負いやすく、使い込むほど役割が混ざりやすい。方向性共有やコンセプト整理は独立したダッシュボードに寄せ、実際の進捗や実装履歴は `CHANGELOG.md` で追う方が、役割分担が明確になりそう。

**改善策・対応**:
将来的には、タスクの状態管理は issue に移し、コンセプト共有や方針確認には別ダッシュボードを使う構成を検討する。`CHANGELOG.md` は実装済みの変更履歴と現在地の把握に使い、各管理手段の責務を重ねすぎないようにする。

## 2026-04-18: タスク管理ファイルと専用スキルは最小限に絞った方が並列開発しやすい

**カテゴリ**: フロー改善

**気づき**:
管理ファイルを増やしてタスクや進捗を複数箇所で持つと、更新コストが増えるだけでなく、ワークツリー運用時にどれが正なのか分かりづらくなりやすい。特に並列開発では git 管理された共有ファイルで進捗が見えないと他エージェントから状況を把握できず、不都合が出る。運用上は、git で共有されるタスク管理表は1個、多くても2個程度に絞る方が効率がよい。また、実装状況や改善の積み上がりは WBS だけでは把握しきれないため、`CHANGELOG.md` で変更履歴を一覧できる状態を作ると、LLM に現在地を素早く渡せてタスク生成のボトルネックを減らしやすい。一方で、ワークフロー専用のスキルや管理ファイルを増やしすぎると、読み込みコンテキスト、説明コスト、運用変更時のメンテナンスコストが重くなり、試行錯誤フェーズでは足かせになりやすい。

**改善策・対応**:
タスク管理は `docs/wbs.md` を中心に据え、変更の流れや細かな改善履歴は `CHANGELOG.md` に集約する運用を優先する。固定ワークフローに強く依存する専用スキルはむやみに増やさず、使い回しが効く最小限のものだけを残す。運用が固まるまでは、管理対象ファイル数とスキル数を意識的に絞り、変更時に追随すべき箇所を少なく保つ。

## 2026-04-17: タスク生成のボトルネックを解消するためにチェンジログで開発文脈を蓄積する

**カテゴリ**: フロー改善

**気づき**:
LLM によるタスク実行速度はすでに十分速い。個人開発での真のボトルネックは「タスクの生成速度」。次に何をすべきかを自分で考えて言語化するコストが、実装コストを上回っている。LLM にタスク生成・プランニング・問題点の指摘をさせたいが、そのためにはプロジェクトがどういう変遷を経て現在に至ったかという文脈が必要。

**改善策・対応**:
`PROJECTMEMORY/CHANGELOG.md` を作成し、タスク完了のたびに「何をやったか・なぜやったか」を時系列で記録する。このファイルを別 LLM へのコンテキストとして渡すことで、タスク生成・リスク指摘・プランニングを LLM に委任できる状態を目指す。

記録単位の目安:
- 完了したタスク名
- 実装した内容の概要（コードの詳細ではなく設計判断・変更の意図）
- 背景・動機（なぜこの変更が必要だったか）
- 残課題・次のステップへのヒント
