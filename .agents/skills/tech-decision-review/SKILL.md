# Tech Decision Review — 技術判断レビュースキル

技術選択・アーキテクチャ判断を受け取り、**トレードオフの可視化**と**失敗シナリオの先回り**の2軸でレビューする。

技術判断は「正解」がないことが多い。だからこそ、選択肢ごとの代償を明示し、「何を捨てているか」を意識した上で決定できるようにする。

---

## 入力

レビュー対象は以下のいずれか:
- ライブラリ・フレームワークの選定
- ディレクトリ構造・モジュール分割の設計
- データモデル・スキーマ設計
- API 設計・通信方式の選択
- 「AとBどっちがいい？」という比較判断
- 「この方針で進めたい」という技術方針

対象が曖昧な場合は、決定のスコープと制約条件を確認してから進む。

---

## 軸1: トレードオフ分析

あらゆる技術判断にはトレードオフがある。何かを得るために何を捨てているかを明示する。

### 評価の観点

| 観点 | 問い |
|------|------|
| **複雑さ** | この選択はコードベースの複雑さをどれだけ増やすか。学習コストは？新メンバーが理解するまでの時間は？ |
| **依存リスク** | 外部依存は増えるか。メンテナンス状況は？ライセンスは？突然の breaking change リスクは？ |
| **移行コスト** | 後から別の選択肢に乗り換えるコストはどれだけか。ロックインの度合いは？ |
| **拡張性** | 将来の要件変更（仕様 v003 の将来実装リスト等）に対してどれだけ柔軟か |
| **パフォーマンス** | ランタイムのパフォーマンスへの影響。バンドルサイズ、メモリ、応答速度 |
| **既存コードとの整合** | 現在のコードベースのパターンや規約と一致するか。異質な要素を持ち込まないか |

### 分析の進め方

1. **選択肢の列挙**: 検討中の選択肢を列挙する。ユーザーが1つしか挙げていなくても、有力な代替案があれば提示する。「何もしない（現状維持）」も選択肢に含める

2. **観点ごとの比較**: 各選択肢を上記の観点で評価する。すべての観点が等しく重要ではない — プロジェクトの現状と制約に応じて重みが変わる

3. **隠れたコストの指摘**: 選択肢の良い面だけでなく、見落としやすい代償を指摘する。ライブラリの便利さの裏にあるバンドルサイズ増、抽象化の裏にあるデバッグの困難さなど

4. **推奨と根拠**: 最終的に1つを推奨する場合は、推奨理由と「この推奨が覆る条件」を明示する

---

## 軸2: 失敗シナリオ分析（プリモーテム）

「この技術判断が裏目に出るとしたら、どう裏目に出るか」を事前に想像する。

### 失敗カテゴリ

**統合の失敗** — 組み合わせで壊れる
- 既存ライブラリ・フレームワークとの互換性問題
- ビルドシステム・ツールチェーンとの相性
- 型システム上の不整合

**スケールの失敗** — 小さいうちは良いが大きくなると破綻する
- データ量が増えた時のパフォーマンス劣化
- ファイル数・モジュール数が増えた時の管理困難
- チームが大きくなった時の認知的オーバーヘッド

**進化の失敗** — 要件変更に耐えられない
- 仕様変更で根本的な作り直しが必要になる
- 「ちょっとした拡張」が異常に難しくなる設計
- 依存先のライブラリがメンテされなくなる

**運用の失敗** — デバッグ・保守が困難
- エラーメッセージが不明瞭で問題の特定が難しい
- 暗黙の動作が多く、挙動の予測が困難
- テストが書きにくい設計

### 分析の進め方

1. **シナリオ列挙**: 各カテゴリから具体的な失敗シナリオを書き出す。「3ヶ月後に○○の要件が追加されたとき、△△の設計では□□が起きる」の形で具体的に

2. **影響度と発生確率**: 各シナリオを「致命的 / 痛い / 軽微」と「高 / 中 / 低」で分類

3. **対策の提案**:
   - **設計変更**: リスクを避ける代替設計
   - **抽象化層の挿入**: 将来の差し替えを容易にする中間層
   - **制約の文書化**: `docs/decisions.md` に判断理由と前提条件を記録し、前提が崩れたときに気づけるようにする
   - **受容**: リスクを認識した上で受け入れる

## 出力フォーマット

```markdown
# 技術判断レビュー: {対象の名前}

## 選択肢の比較

| 観点 | 選択肢A | 選択肢B | 現状維持 |
|------|---------|---------|----------|
| 複雑さ | ... | ... | ... |
| 依存リスク | ... | ... | ... |
| ... | ... | ... | ... |

## 推奨
（推奨する選択肢とその根拠。推奨が覆る条件も明記）

## 失敗シナリオ

### 最も警戒すべきシナリオ
（致命的 × 高確率。具体的な対策付き）

### 注意すべきシナリオ
（痛い × 中確率）

### 許容できるリスク
（軽微 or 低確率）

## decisions.md への記録案
（判断理由と前提条件の要約。そのままコピーして使える形式）
```

---

## 使い方のバリエーション

- **ライブラリ選定**: 「状態管理に Zustand と Jotai どっちがいい？」→ 比較表 + 失敗シナリオ
- **設計方針の検討**: 「ストアを分割すべきか一枚岩のままか」→ トレードオフ + スケールの失敗分析
- **既存判断の振り返り**: 「Lexical を選んだ判断は今も妥当か」→ 前提条件の変化を確認
- **移行判断**: 「○○から△△に移行すべきか」→ 移行コスト + 移行しない場合の失敗シナリオ
