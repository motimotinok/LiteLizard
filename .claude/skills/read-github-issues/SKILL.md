# Read GitHub Issues

GitHub Issue から LiteLizard の次タスク候補を探す。

このスキルはユーザーが明示的に Issue 確認を依頼したときだけ使う。通常の「次のタスクどれ？」では、まず `docs/wbs.md`、関連仕様、実装済みコードを確認する。

## 使う場面

- 「Issueから次タスクを探して」
- 「Issueを見て優先度順に候補出して」
- 「GitHub Issueから作業を選んで」
- 「open Issueを確認して」

使わない場面:

- ユーザーが Issue と言っていない通常のタスク選定
- 現在の WBS タスクだけを確認したい場合
- Issue 作成依頼（その場合は `create-github-issue`）

## 標準フィルター

標準では以下を見る。

- open Issue
- `blocked` ラベル付きは通常候補から除外
- `in-progress` ラベル付きは別エージェント/人間が着手中のため通常候補から除外（状況確認だけ簡潔に報告）
- 優先順は `P0` → `P1` → `P2`
- `agent-ready` ラベル付きは自律エージェントに任せられるため、人間視点の候補としては **後回し**。優先度が同じなら `agent-ready` が付いていないものを先に出す

基本コマンド:

```bash
gh issue list --state open --json number,title,labels,updatedAt,url
```

本文確認が必要な候補は個別に読む。

```bash
gh issue view <number> --json number,title,body,labels,state,updatedAt,url
```

## 候補選定の流れ

1. open Issue 一覧を取得する
2. ラベルで `blocked` と `in-progress` を通常候補から外す
3. `P0`、`P1`、`P2` の順に並べ、同優先度内では `agent-ready` が **付いていないもの** を先に出す（agent-ready は自律エージェントに任せ、人間の時間は agent-ready 化されていない Issue に充てる）
4. 候補Issueの本文を読み、背景・意図・ゴール・スコープ・非ゴール・受け入れ条件・検証方法・Depends on を確認する
5. `Depends on:` がある場合、依存先がすべて close または PR マージ済みかを確認し、未解消なら「依存待ち」に分類する
6. `in-progress` 付きの Issue があれば、変更領域が新規候補と重なるか LLM 判断し、重なる場合は新規候補を「衝突回避で待機」に分類する
7. 必要に応じて `docs/wbs.md`、関連 `docs/specs/*.md`、`docs/decisions.md` を確認する
8. 実装済みの可能性がある場合は、関連コードと最近の `git log` を確認する
9. Issue が1回の作業単位として扱えるか、検証可能か、スコープが広すぎないかを評価する
10. 候補を「今やる候補」「まだ早い候補」「ブロック中/確認必要」「衝突回避で待機」に分ける

ラベル運用と衝突判定の詳細は `docs/agent-flow.md` を参照。

Issue だけで実装判断を確定しない。Issue は古くなるため、WBS・仕様・コードで現在地を確認する。

## 本文の読み方

良い候補は、実装手順が細かい Issue ではなく、意図・制約・検収条件が明確な Issue。

優先して確認すること:

- ゴールが1つに絞られている
- 受け入れ条件が観測可能である
- 検証方法が書ける
- スコープと非ゴールがあり、作業範囲を広げすぎずに済む
- 既存仕様を壊さないための制約が書かれている

注意すること:

- 「いい感じに改善」「全体的に整理」など、完了判断が曖昧な Issue は確認必要に回す
- 手順が細かく書かれていても、実装時は手順に盲従しない
- Issue の背景・意図・ゴール・非ゴール・受け入れ条件・検証方法を優先し、既存コード調査後に最小変更で方針を判断する
- 読んでいないコードについて推測で断定しない

## 出力形式

日本語で短く出す。

```markdown
おすすめは #123「タイトル」です。

理由:
- P1 で blocked ではない
- WBS の L-09 と対応している
- ゴールと受け入れ条件が観測可能
- 非ゴールが明確で、作業範囲を広げずに進められる

実装前に確認するもの:
- docs/wbs.md の L-09
- docs/specs/...
- 関連コードの現在の実装

次点:
- #124「タイトル」: P2 なので後回しでよい

確認が必要:
- #125「タイトル」: 検証方法が不足しており、完了判断が曖昧です
```

## 判断ルール

- `P0`: 最優先。ただし再現不能なバグは確認が必要
- `P1`: MVP/公開に必要なら強い候補
- `P2`: 価値があっても、P1 が残っているなら通常は後回し
- `blocked`: 標準候補から外し、ブロック理由だけ報告する
- `in-progress`: 標準候補から外し、誰がいつ着手したか（PR があれば PR 番号）だけ簡潔に報告する
- `agent-ready`: 自律エージェントに任せられるため、人間視点の候補としては後回し。同優先度なら付いていないものを上に並べる
- ラベルが不足している Issue は、本文から推定してよいが、推定であることを明記する
- ラベルが強くても、ゴール・受け入れ条件・検証方法が曖昧なら確認必要に回す
- 既に実装済みの可能性がある Issue は、コードと最近の `git log` で確認してから候補にする

## WBS との関係

- Issueに WBS ID がある場合は、その行を確認する
- WBSに未反映のIssueでも、ユーザーが明示しない限り WBS 追加候補として扱わない
- WBS への新規タスク追加ではなく、GitHub Issue を次タスク選定の主な入口にする
- このスキルでは WBS を自動更新しない

## Issue の close タイミング

Issue は、Codex 側の実装完了だけでは close しない。

原則として以下が揃ってから close する。

- 実装が完了している
- 必要なローカル検証が通っている
- 変更が commit / push されている
- GitHub CI が通っている
- ユーザー確認が完了している、またはユーザーが明示的に close を依頼している

ローカル修正済みでも、ユーザー確認前・CI 前の Issue は「実装済み、close 待ち」として扱う。

Issue 作業の完了報告では、最後に短く一文だけ「確認してOKなら close 指示をください」とユーザーへ伝える。
