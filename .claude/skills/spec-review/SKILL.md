# Spec Review — 仕様レビュースキル

仕様・機能案・設計方針を受け取り、**人間の本能への訴求**と**失敗シナリオの先回り**の2軸でレビューする。

このスキルの価値は「作る前に気づく」こと。実装してから「誰も使わなかった」「想定外の使われ方で壊れた」を防ぐために、仕様段階で需要と脆弱性を可視化する。

---

## 入力

レビュー対象は以下のいずれか:
- 仕様書・設計ドキュメント（ファイルパスまたは会話中の説明）
- 機能案・ユーザーストーリー
- 「こういう方向で進めたい」という方針

対象が曖昧な場合は、まず「何をレビューするか」を確認してから進む。

---

## 軸1: 本能分析

人はプロダクトを「便利だから」使うのではなく、自分の中にある根源的な欲求が満たされるから使い続ける。この軸では、仕様が人間の8つの本能のどこに刺さっているかを分析し、まだ活かせていない本能への拡張ポイントを提案する。

### 8つの本能

| 本能 | 核心 | プロダクトでの現れ方（例） |
|------|------|--------------------------|
| **安らぐ** | 不安の除去、安心感 | 自動保存、データが消えない安心、操作の取り消し |
| **進める** | 前進・成長の実感 | 進捗バー、文字数カウント、「ここまで書けた」の可視化 |
| **決する** | 自分で選び取る主体性 | モード切り替え、カスタマイズ、「自分のやり方」で使える |
| **有する** | 所有・蓄積の満足 | 作品が自分のものとして残る、コレクション感、ファイル管理 |
| **属する** | つながり・共同体感 | 共同編集、コミュニティ、「同じツールを使う仲間」意識 |
| **高める** | 自己効力感・上達 | 分析フィードバック、文章力の向上を実感できる指標 |
| **伝える** | 表現・発信の欲求 | エクスポート、共有機能、人に見せられる形にする |
| **物語る** | 意味づけ・文脈の構築 | 執筆の流れが見える、章構成の俯瞰、創作プロセスの記録 |

### 分析の進め方

1. **現状マッピング**: 対象の仕様が8つの本能のどれに訴求しているかを判定する。強く刺さっているもの、弱く触れているもの、まったく触れていないものを分類する

2. **訴求の深さを評価**: 単に「触れている」のと「深く満たしている」のは違う。例えば自動保存は「安らぐ」に触れているが、「保存された安心」を能動的に伝える UI があるかどうかで深さが変わる

3. **拡張ポイントの提案**: まだ活かせていない本能について、自然な形で仕様に組み込める可能性を提案する。無理に全部カバーする必要はない — プロダクトの性質と相性の良い本能に絞る

4. **優先度の判断**: 既存の強みを深めるべきか、新たな本能を開拓すべきか。一般に、既存の強みを深めるほうがコストが低く効果が高い

## 軸2: 失敗シナリオ分析（プリモーテム）

「この仕様が失敗するとしたら、どう失敗するか」を事前に想像し、対策を立てる。楽観的に「うまくいくはず」と進めるのではなく、失敗のパターンを先に洗い出しておくことで、致命傷を避けられる。

### 失敗カテゴリ

以下の観点から失敗シナリオを検討する:

**需要の失敗** — 作ったが誰も使わない
- ターゲットユーザーが本当にこの問題を抱えているか
- 既存の代替手段（他ツール、手作業）で十分満足していないか
- 「あると嬉しい」と「ないと困る」の区別

**体験の失敗** — 使い始めたが続かない
- 初回体験で価値が伝わるか（Time to Value）
- 操作の学習コストと得られる価値のバランス
- 「わからない」「面倒」で離脱するポイント

**技術の失敗** — 作れない・壊れる
- 実装の複雑さが想定以上になるリスク
- パフォーマンス・スケーラビリティの問題
- 既存機能との干渉・退行

**スコープの失敗** — 広げすぎ・狭すぎ
- 仕様が膨らんで出荷できない
- 逆に、最小限すぎて価値が伝わらない
- 中途半端な状態でリリースして印象を損なう

### 分析の進め方

1. **シナリオ列挙**: 各カテゴリから最低1つ、具体的な失敗シナリオを書き出す。「ユーザーが○○しようとしたとき、△△が起きて、結果として□□になる」の形で具体的に

2. **影響度と発生確率**: 各シナリオを「致命的 / 痛い / 軽微」と「高 / 中 / 低」で分類する

3. **対策の提案**: 致命的 × 高確率のものから順に対策を提案する。対策は以下のいずれか:
   - **仕様変更**: リスクを根本的に回避する設計変更
   - **段階的リリース**: まず小さく試して検証する
   - **フォールバック**: 失敗した場合の退路を用意する
   - **受容**: リスクを認識した上で受け入れる（低影響のもの）

## 出力フォーマット

以下の構造で出力する。分量は対象の規模に応じて調整する — 小さな機能なら簡潔に、大きな仕様なら詳細に。

```markdown
# 仕様レビュー: {対象の名前}

## 本能分析

### 現状の訴求マップ
（8つの本能それぞれについて、強/弱/なし で分類）

### 深掘りポイント
（既に刺さっている本能をさらに強化する提案）

### 拡張の可能性
（まだ活かせていない本能への自然な拡張案）

## 失敗シナリオ

### 最も警戒すべきシナリオ
（致命的 × 高確率のもの。具体的な対策付き）

### 注意すべきシナリオ
（痛い × 中確率のもの）

### 許容できるリスク
（軽微 or 低確率で、受容判断したもの）

## 総合所見
（2-3行で、この仕様の最大の強みと最大のリスクをまとめる）
```

---

## 使い方のバリエーション

- **仕様全体のレビュー**: 仕様書を読み込んで全体を分析する
- **機能単体のレビュー**: 「この機能を追加したいんだけど」に対して2軸で検討する
- **方針の壁打ち**: 「こっちの方向とあっちの方向、どちらがいいか」を本能訴求と失敗リスクの観点で比較する
- **既存機能の振り返り**: 実装済み機能を改めてレビューし、改善点を見つける
