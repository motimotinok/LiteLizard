# Update Product Map

`docs/product-map.md` を、LiteLizard の全体像・現在地・ロードマップ・チケット化前の検討テーマの地図として更新する。

## 使う場面

- 思いつきや将来構想を、まだ実装チケットにせず整理したい
- 現在地（直近の重心、未完了チケットの状態）を最新化したい
- ロードマップを Now / Next / Later で動かしたい（Next を Now に上げる、Later を Next に降ろす、新規テーマを追加する など）
- 次にやる候補や人間が判断するテーマを product-map に反映したい
- GitHub Issue や会話中の構想を product-map に要約したい
- product-map が古くなったので、現状に合わせて棚卸ししたい

使わない場面:

- すぐ実装できる作業を作る場合（`create-ralph-ticket`）
- 完了した作業を履歴に残す場合（`update-changelog`）
- 確定仕様を詳細化する場合（`spec-writer`）
- GitHub Issue を新規作成する場合（`create-github-issue`）

## 成果物

- `docs/product-map.md`

必要がなければ他のファイルは更新しない。

## 成功条件

- product-map が、現在の開発方針・現在地・ロードマップ・判断テーマを短く表している
- 「現在地」セクションが、直近の重心と未完了チケットの状態を 1〜2 文で示している
- ロードマップが Now / Next / Later の 3 段で書かれており、各テーマに**前提となる判断**が 1 行添えられている
- 細かい進捗、完了履歴、実装手順が混ざっていない
- 実装に移すべきもの（Now）と、まだ判断が必要なもの（Next / Later / 判断テーマ）が区別されている
- `docs/tickets/`、`CHANGELOG.md`、`docs/specs/`、`docs/decisions.md` との役割が重複していない
- 不要な全面書き換えや文体統一だけの差分が入っていない

## 制約

- product-map を WBS やタスク台帳にしない
- チケット単位の status、担当、期限、完了チェックを持たせない
- 完了履歴は `CHANGELOG.md` に任せる
- 実装キューは `docs/tickets/` に任せる
- 確定仕様の詳細は `docs/specs/` と `docs/decisions.md` に任せる
- GitHub Issue を現役実装キューとして扱わない
- 曖昧な構想を勝手に実装チケット化しない
- Now を勝手に増やさない（人間または明示的な依頼が必要）

## 構成

既存構成をできるだけ保ち、必要なセクションだけ最小差分で更新する。基本の置き場所は次の通り。

- **プロダクトの核**: 変えない前提の大きな価値・体験
- **現在の運用**: チケット運用・退役済み資料の扱い
- **現在地**: 直近の重心と未完了チケットの状態を 1〜2 文で示す
- **ロードマップ**: Now / Next / Later の 3 段
  - **Now**: 直近のチケット数本で扱う対象。空でもよい（その場合は「未確定」と書く）
  - **Next**: 方向は固まりつつあるが、設計判断や前提が片付いてから Now に上げるテーマ
  - **Later**: 構想段階。GitHub Issue がある場合は `#番号` を添える
  - 各テーマには「**前提**」を 1 行添えて、判断テーマとの依存関係を見える化する
- **成熟している領域**: 背景として参照される完了済みの大きな機能領域
- **人間が判断するテーマ**: 判断が解けると Next → Now に動くもの。各項目に対応するロードマップ項目を `（→ Next: ...）` の形で添える
- **チケット化の基準**: 運用ルール
- **関連スキル**: `update-product-map` / `create-ralph-ticket` / `update-changelog` の役割分担

## 思いつき・依頼の振り分け

- すぐ実装できるなら product-map ではなく `create-ralph-ticket` を提案する
- 方向は決まったが前提判断が残るなら **Next** に短く入れる
- 構想段階なら **Later** または「人間が判断するテーマ」に入れる
- 外部参照や長期構想として残す必要があるなら GitHub Issue 化を提案し、Later に `#番号` を添える
- 「これを次にやる」と人間が決めたものだけ **Now** に上げる

## ロードマップを動かす操作

- **Next → Now**: 前提が解けて実装範囲が見えたとき。Now に移したらすぐ `create-ralph-ticket` を提案する
- **Later → Next**: 方向が固まり、前提となる判断テーマが具体化したとき
- **新規 → Later/Next/判断テーマ**: 思いつきを置く先を、確度に応じて選ぶ
- **削除**: 方針が変わって追わないと決めたテーマは、削除する旨を `CHANGELOG.md` に残してから product-map から外す

Now / Next / Later の段を移したら、依存関係（前提となる判断）も合わせて更新する。

## 完了時の報告

更新したセクションと、product-map に置いた理由を短く報告する。Now を動かした場合は、続けて `create-ralph-ticket` を提案するかどうかも示す。チケット化した方がよい具体項目が見つかった場合は、勝手に作らず候補として示す。
