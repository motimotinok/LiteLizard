# Prepare Agent-Ready Issues

既存 GitHub Issue を、自律 LLM エージェントが拾って PR まで進められる状態か診断し、必要な整理案を出す。

このスキルは「実装する Issue を選ぶ」ためではなく、**Issue を agent-ready ラベル付与前の状態まで整える**ために使う。最終的に `agent-ready` を付けるかどうかはユーザー判断に残す。

## 使う場面

- 「agent-ready にできる Issue を整理して」
- 「既存 Issue に agent-ready を付けられるか見て」
- 「自律実行に回せるよう Issue を整えて」
- 「Issue の不足情報を確認して」
- 「この Issue を LLM に任せられる状態にしたい」

使わない場面:

- 次に実装する Issue を1件選ぶだけの場合（`read-github-issues`）
- 新しい Issue を作る場合（`create-github-issue`）
- 実装に着手する場合
- WBS / CHANGELOG の完了反映だけの場合（`update-wbs-changelog`）
- 仕様策定・設計判断そのものを目的にした Issue
- `agent-ready` 化するための仕様明確化だけで重い作業になる Issue

## ゴール

対象 Issue について、以下を日本語で出す。

- `agent-ready` を付けてよいか
- 足りない情報は何か
- 足りない仕様についての具体的な提案
- ユーザーに確認すべき質問
- Issue 本文へ追記する案
- 必要なら、分割案・`blocked` 候補・実装済み疑い

## 重要な制約

- 勝手に `agent-ready` ラベルを付けない
- 勝手に `blocked` ラベルを付けない
- 勝手に Issue を close しない
- 勝手に WBS を更新しない
- 勝手に実装へ進まない
- Issue 本文の更新やラベル操作は、ユーザーが明示的に依頼した場合だけ行う
- 仕様策定そのものをこのスキルで進めない
- 重い仕様明確化が必要な Issue は、追記案作成まで踏み込まず「対象外」として短く理由を報告する

`agent-ready` は「LLM ができそう」ではなく、「人間が仕様・受け入れ条件・依存関係を確認し、自律実行に渡してよいと判断した」状態として扱う。

## 対象外にする Issue

以下に当てはまる Issue は、このスキルでは深掘りしない。`agent-ready` 化の候補から外し、理由だけ短く報告する。

- 仕様策定・設計検討そのものがゴールになっている
- 選択肢の比較、プロダクト判断、UX 方針決定が主作業になっている
- 受け入れ条件を作る前に、まず機能の意味・対象ユーザー・成功条件を決める必要がある
- 既存仕様との整合性確認が大きく、Issue 整備だけで独立した設計タスクになる
- 複数の実装案があり、どれを採るかで将来の構造や運用が大きく変わる
- Issue を LLM に処理可能にするための質問・提案・確認が多くなりすぎる

このスキルで扱うのは、**不明点を数点詰めれば自律実行に渡せそうな Issue** に限る。仕様そのものを考える Issue は、人間との仕様策定タスクとして別に扱う。

## 確認する資料

対象 Issue ごとに、必要な範囲で確認する。

- Issue 本文・ラベル・コメント
- `docs/agent-flow.md`
- `prompts/agent-pickup.md`
- `docs/wbs.md`
- `CHANGELOG.md`
- 関連する `docs/specs/*.md`
- `docs/decisions.md`
- 関連する実装済みコード
- 最近の `git log`（実装済み・方針変更済みの疑いがある場合）

Issue は古くなるため、本文だけで判断しない。必ず現在の仕様・コード・履歴と照合する。

## 基本コマンド

Issue 一覧:

```bash
gh issue list --state open --json number,title,labels,updatedAt,url
```

対象 Issue の本文:

```bash
gh issue view <number> --json number,title,body,labels,state,updatedAt,url,comments
```

関連ラベル確認:

```bash
gh label list
```

## 診断フロー

1. 対象 Issue を決める
   - ユーザーが番号を指定した場合はその Issue を見る
   - 指定がない場合は open Issue 一覧を取得し、`agent-ready` が付いていないものを中心に見る
   - `in-progress` は通常対象外。ただし「整理だけ」なら状態確認として扱う
   - 仕様策定そのもの・重い仕様明確化が必要な Issue は、この時点で対象外候補にする

2. Issue 本文を読む
   - 背景
   - 意図
   - ゴール
   - スコープ
   - 非ゴール
   - 受け入れ条件
   - 検証方法
   - 依存関係

3. 現在の repo と照合する
   - WBS ID があれば `docs/wbs.md` を確認する
   - 関連仕様・decision を確認する
   - 関連コードを確認し、既に実装済みでないか見る
   - 似た Issue や最近の変更と重複しないか確認する

4. `agent-ready` 可否を判定する
   - 可: 追加質問なしで自律エージェントが PR まで進められそう
   - 条件付き可: Issue 本文へ追記すれば渡せそう
   - 不可: 仕様判断、分割、依存解消、人間レビューが先
   - 対象外: 仕様策定そのもの、または agent-ready 化のための仕様明確化が重すぎる

5. 不足情報を整理する
   - 仕様不足
   - 受け入れ条件不足
   - 検証方法不足
   - 非ゴール不足
   - 依存関係不足
   - スコープ過大
   - 実装済み疑い

6. 仕様不足は、必ず提案つきで確認する
   - 空欄をそのまま質問しない
   - 既存仕様・コード・ユーザーの過去方針から最も自然な案を出す
   - 代替案がある場合は、主案とその理由を短く書く
   - ユーザーには「この案でよいか」を確認する
   - ただし、提案を作るための仕様検討が重い場合は提案作成に進まず、対象外として扱う

## `agent-ready` 判定基準

付けてよい条件:

- ゴールが1つに絞られている
- 受け入れ条件が観測可能である
- 検証方法が書ける
- スコープと非ゴールが明確である
- 依存 Issue があれば `Depends on` が明記されている
- 依存が未解決なら `blocked` または待機扱いにできる
- 既存仕様・コードと矛盾していない
- 既に実装済みではない
- 1 PR で完了できる大きさである
- LLM が「いい感じに判断」する余地が少ない

まだ付けない条件:

- 「いい感じに改善」「全体的に整理」など完了判断が曖昧
- UI/UX の好み判断が主で、人間の確認が必要
- 設計選択肢が複数あり、どれを採るかで将来構造が変わる
- 受け入れ条件が観測できない
- 検証方法がない
- 依存関係や実装順が不明
- 複数の作業が混ざっている
- 既存コードと照合すると、Issue 内容が古い・実装済み・重複の疑いがある

対象外にする条件:

- Issue の目的が実装ではなく仕様策定・設計検討である
- 不明点を詰める作業自体が重く、Issue 整備ではなく別の仕様策定タスクになっている
- LLM に任せられる粒度にするために、プロダクト判断や将来設計の大きな決定が必要である

判断の短い基準:

> 初見の自律エージェントに渡して、追加質問なしで PR まで行けるか。

もう1つの基準:

> 数点の確認で agent-ready にできない Issue は、このスキルの対象ではない。

## 仕様不足時の質問方針

不足情報がある場合は、質問だけで止めず、必ず提案を添える。

良い聞き方:

```markdown
不足: 保存失敗時の扱いが未定義です。

提案:
今回は既存の保存フローに合わせて、保存失敗時は処理を中断し、既存データを変更せず、ユーザーにエラーを表示する方針がよさそうです。リトライ UI や詳細ログ出力は非ゴールにします。

確認:
この方針で Issue 本文に追記してよいですか？
```

避ける聞き方:

```markdown
保存失敗時はどうしますか？
```

複数案がある場合:

```markdown
提案A（推奨）: 既存仕様維持で最小変更にする
提案B: 新しい設定項目を追加する

推奨はAです。今回の Issue は自律実行向けに小さく閉じたいので、設定追加は別 Issue に分ける方が安全です。
```

## 出力形式

対象が1件の場合:

```markdown
#123「タイトル」は、現時点では 条件付きで agent-ready 候補 です。

理由:
- ゴールは明確
- 受け入れ条件は一部不足
- 検証方法は追加が必要
- 関連コードを見る限り未実装

不足:
- エラー時の期待動作が未定義
- 非ゴールに「保存形式の変更なし」を明記した方がよい

提案:
- エラー時は既存データを変更せず、ユーザーに失敗を通知する
- 今回は保存形式・同期処理・UI 全体 redesign は非ゴールにする

確認したいこと:
- 上の方針で Issue 本文に追記してよいですか？

Issue 本文追記案:
...
```

複数件を見る場合:

```markdown
agent-ready 候補:
- #123「タイトル」: 受け入れ条件を1点追記すれば可

要整理:
- #124「タイトル」: スコープが広いため分割案を提示
- #125「タイトル」: 仕様不足。提案Aで進めてよいか確認が必要

対象外:
- #128「タイトル」: 仕様策定そのものが主作業で、agent-ready 化のための整理が重すぎる

blocked 候補:
- #126「タイトル」: #120 のマージ待ち

実装済み疑い:
- #127「タイトル」: 現在のコードでは対応済みに見えるため close 判断が必要
```

## Issue 本文追記案の作り方

追記案は、`create-github-issue` の標準構成に寄せる。

- 背景
- 意図
- ゴール
- スコープ
- 非ゴール
- 受け入れ条件
- 検証方法
- Depends on（必要な場合）
- 関連（WBS / Spec / Decision / ファイル）

既存 Issue 本文を全面置換する必要はない。差分として追記しやすい形で出す。

## ラベル操作

ユーザーが「ラベルを付けて」と明示した場合だけ、以下を実行する。

- `agent-ready` を付ける前に、Issue 本文が判定基準を満たしていることを再確認する
- `blocked` が付いている Issue には `agent-ready` を付けない
- `agent-ready` と `blocked` を同時に付ける必要がある場合は、原則として `blocked` を優先し、ユーザーに確認する
- ラベルが存在しない場合は勝手に作らず、ユーザーに確認する

## 完了時の報告

最後に、次の一手を短く示す。

- 「この追記案でよければ Issue 本文を更新できます」
- 「確認事項に OK なら `agent-ready` 付与候補にできます」
- 「これは分割後に再診断するのがよさそうです」

Issue の close が関係する場合は、作業完了後に close するかどうかをユーザーに確認する。
