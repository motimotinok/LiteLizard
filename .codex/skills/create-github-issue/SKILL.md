# Create GitHub Issue

LiteLizard の会話中に出た将来タスク、残課題、バグを GitHub Issue にする。

Issue は「後で拾う作業カード」。`docs/wbs.md` はプロジェクト台帳、`CHANGELOG.md` は完了後の履歴として分ける。このスキルでは WBS を自動更新しない。

## 使う場面

- ユーザーが「Issueにして」「GitHub Issueを作って」と依頼した
- 今すぐ実装しないが、背景つきで残したい
- 別エージェントが後から読んで作業できる粒度にしたい
- バグの再現条件や完了条件を GitHub 上に残したい

使わない場面:

- その場で直す小修正
- WBS/CHANGELOG 更新だけ
- 仕様が曖昧すぎて作業カード化できないメモ

## ラベル

使うラベルは最小限にする。

優先度:

- `P0`: 作業が止まる、または重大な破壊
- `P1`: MVP/公開に必要
- `P2`: 将来改善、UX向上、後でよい

種別:

- `bug`: 期待動作と違う
- `feature`: 新機能、設定追加、ユーザー向け挙動追加
- `docs`: 仕様、WBS、CHANGELOG、運用文書
- `refactor`: 振る舞いを変えない内部整理

状態:

- `blocked`: 依存タスク未完了、仕様未決定、外部要因待ち

ラベルが未作成の場合、勝手に作らない。不足ラベルをユーザーに伝え、作成するなら確認してから行う。

## 手順

1. `gh label list` で既存ラベルを確認する
2. Issue のタイトル、本文、ラベル候補を作る
3. 不足ラベルがある場合は、Issue本文に不足ラベルを注記するか、ユーザーに作成確認を取る
4. ユーザーが内容確認を求めている場合は、発行前にタイトル・本文・ラベルを提示する
5. 発行する場合は `gh issue create` を使う
6. 発行後は Issue URL と付けたラベルを報告する

## Issue 本文テンプレート

```markdown
## 背景

なぜ必要か。UX、バグ、仕様判断、残課題など。

## やること

- 実装・調査・文書化する対象
- 変更する主な領域

## やらないこと

- 今回のIssueに含めない範囲
- 後続タスクに回す内容

## 完了条件

- 期待動作
- 必要なテスト
- 必要なドキュメント更新

## 関連

- WBS候補: なし / L-09 など
- Spec: docs/specs/...
- Decision: docs/decisions.md
- 関連ファイル: path/to/file
```

## タイトルの付け方

- WBS ID がある場合: `L-09: 分析コンテキストポリシー切替`
- WBS ID がない場合: `分析カードの履歴切替を改善する`
- バグの場合: `bug: 章削除後に解析履歴が古い世代へ残る`

短く、検索できる語を入れる。説明は本文に寄せる。

## WBS との関係

- このスキルでは `docs/wbs.md` を更新しない
- WBS に載せる必要がありそうな場合は、Issue本文の `WBS候補` に書く
- ユーザーが明示的に WBS 更新も依頼した場合だけ、別途適切なスキルで対応する
