# Create GitHub Issue

LiteLizard の会話中に出た将来構想、検討メモ、外部参照を GitHub Issue にする。

現役の実装キューは `docs/tickets/` の Ralph Loop チケット。GitHub Issue は、実装キューではなく、将来検討や外部参照として残す場合に使う。このスキルでは WBS を更新しない。

Issue には細かい実装手順ではなく、意図・制約・ゴール・検収条件を書く。LLM エージェントには既存コードを読ませたうえで実装判断を任せる。

実装可能な作業としてすぐ回すものは、原則として GitHub Issue ではなく Ralph Loop チケットにする。Issue 化はユーザーが明示した場合だけ行う。

## 使う場面

- ユーザーが「Issueにして」「GitHub Issueを作って」と依頼した
- 今すぐ実装しないが、背景つきで残したい
- 別エージェントが後から読んで作業できる粒度にしたい
- バグの再現条件や完了条件を GitHub 上に残したい

使わない場面:

- その場で直す小修正
- CHANGELOG 更新だけ
- 仕様が曖昧すぎて作業カード化できないメモ

## 基本方針

Issue に書くこと:

- なぜこの Issue が必要なのか
- 何ができれば完了なのか
- どこまで触ってよいのか
- 何をしてはいけないのか
- どう検証すれば完了と判断できるのか

原則として書きすぎないこと:

- どのファイルをどう書き換えるか
- どの関数をどう分割するか
- どの順番で実装するか
- 人間が事前に想像した細かい修正手順

ただし、既に調査済みで変更箇所が明確な場合や、絶対に守るべき実装上の制約がある場合は書いてよい。

## 粒度

Issue は LLM エージェントが1回の作業単位として扱える大きさにする。

よい Issue:

- ゴールが1つに絞られている
- 完了条件が観測可能である
- 検証方法が書ける
- 変更範囲が広がりすぎない
- 「ついでにやること」が混ざっていない

悪い Issue:

- エディタ周りをいい感じに改善する
- UIを全体的に整える
- 保存処理を全部見直す
- ついでにテストも整理する
- 気になるところをリファクタする

## ラベル

使うラベルは最小限にする。

優先度:

- `P0`: 作業が止まる、または重大な破壊
- `P1`: MVP/公開に必要
- `P2`: 将来改善、UX向上、後でよい

種別:

- `bug`: 期待動作と違う
- `feature`: 新機能、設定追加、ユーザー向け挙動追加
- `docs`: 仕様、CHANGELOG、運用文書
- `refactor`: 振る舞いを変えない内部整理

状態:

- `blocked`: 依存タスク未完了、仕様未決定、外部要因待ち

ラベルが未作成の場合、勝手に作らない。不足ラベルをユーザーに伝え、作成するなら確認してから行う。

## 作成時の固定操作

1. `gh label list` で既存ラベルを確認する
2. Issue のタイトル、本文、ラベル候補を作る
3. 不足ラベルがある場合は、Issue本文に不足ラベルを注記するか、ユーザーに作成確認を取る
4. ユーザーが内容確認を求めている場合は、発行前にタイトル・本文・ラベルを提示する
5. 発行する場合は `gh issue create` を使う
6. 発行後は Issue URL と付けたラベルを報告する

## Issue 本文テンプレート

```markdown
## 背景

このIssueが必要になった理由を書く。

現在どのような問題・不便・未実装・不整合があるのかを説明する。単に「これを実装する」ではなく、なぜそれが必要なのかを書く。

## 意図

このIssueで実現したい設計上・体験上の狙いを書く。

ユーザー体験、保守性、将来の拡張、既存仕様との整合性など、実装判断の軸になる情報を記述する。

## ゴール

このIssueが完了したとき、ユーザーまたは開発者の視点で何ができるようになっているかを書く。

ゴールは、なるべく観測可能な状態で書く。

- ユーザーが〇〇を実行できる
- 〇〇したときに△△が表示される
- アプリを再起動しても〇〇が保持される
- 既存の〇〇機能を壊さずに△△が追加される

## スコープ

このIssueで扱う範囲を書く。

必要に応じて、対象画面・対象機能・関連しそうな領域を示す。ただし、具体的な実装ファイルや修正手順は、確信がある場合を除いて固定しない。

## 非ゴール

このIssueではやらないことを書く。

- UI全体のリデザインはしない
- 関係ないリファクタリングはしない
- 既存の保存形式は変更しない
- 既存仕様を破壊する変更はしない
- 今回の目的に直接関係しない型修正・lint修正には踏み込まない
- 大規模な設計変更はしない

## 受け入れ条件

このIssueを完了と判断する条件を書く。できるだけチェックリスト形式にする。

- [ ] 〇〇したときに△△になる
- [ ] 空データの場合でもクラッシュしない
- [ ] 既存の〇〇機能がこれまで通り動く
- [ ] エラー時に最低限のフォールバックがある
- [ ] 不要な大規模リファクタリングが含まれていない

## 検証方法

完了後に実行する確認方法を書く。自動テスト、ビルド、lint、手動確認の手順を分けて書く。

- `pnpm test`
- `pnpm build`
- 対象画面を開き、〇〇操作をして△△になることを確認する
- アプリを再起動し、変更後の状態が維持されることを確認する

## 実装方針

具体的な手順は固定しない。

エージェントは既存コードを調査し、既存設計に沿って最小変更で実装すること。実装前に、関連するファイル・既存仕様・呼び出し関係を確認すること。読んでいないコードについて推測で断定しないこと。

迷った場合は、以下を優先する。

1. 既存仕様を壊さない
2. 変更範囲を小さく保つ
3. 将来の大改修より、現在のIssueの完了を優先する
4. 不明点を勝手に大きく解釈しない
5. 必要以上に抽象化しない

## 完了時の報告

作業完了時には、以下を簡潔に報告する。

- 何を変更したか
- どの受け入れ条件を満たしたか
- 実行した検証コマンド
- 検証結果
- 残った懸念や未対応事項がある場合はそれ

## 関連

- 関連チケットや仕様: 既存IDがある場合のみ記載
- Spec: docs/specs/...
- Decision: docs/decisions.md
- 関連ファイル: path/to/file
```

## タイトルの付け方

- 関連チケットがある場合: `docs/tickets/done/2026-05-06-analysis-context-policy.md`
- 関連チケットがない場合: `分析カードの履歴切替を改善する`
- バグの場合: `bug: 章削除後に解析履歴が古い世代へ残る`

短く、検索できる語を入れる。説明は本文に寄せる。

## WBS との関係

- WBS は `docs/old/wbs.md` に退役済み資料として残すだけで、更新しない
- 新規タスクを WBS 候補として書かない
- 実装可能な作業へ落とす場合は、別途 `docs/tickets/` の Ralph Loop チケットを作る
