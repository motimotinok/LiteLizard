# Update WBS Changelog

完了したタスクを `docs/wbs.md` と `CHANGELOG.md` に反映する。

このスキルのゴールは、タスク台帳と変更履歴を同期し、後から「何が完了し、なぜ履歴に残ったか」を追える状態にすること。

## 成果物

- `docs/wbs.md`: 対象タスクの完了状態と必要な完了メモ
- `CHANGELOG.md`: 先頭側に追加する2行の変更履歴

上記2ファイルを1セットで扱う。片方だけ更新して終わらない。

## 成功条件

- WBS の対象タスクが `✅` になっている
- WBS 冒頭の `最終更新` がある場合は当日の日付になっている
- `CHANGELOG.md` の先頭側に今回の2行が追加されている
- どちらのファイルにも不要な大規模整形や無関係な変更が入っていない
- 完了扱いにした範囲と履歴の要約が一致している

## 制約

- 関係ないタスクの並び替えをしない
- 担当、ブランチ、優先度を勝手に再設計しない
- `TASKS.md` や他の台帳に同じ内容を転記しない
- 過去ログの文体統一だけを目的に全面書き換えしない
- タスク ID、完了範囲、要約のいずれかが曖昧なときだけユーザーに確認する

## WBS の更新方針

最小差分で反映する。

- 対象タスクの状態を `✅` に更新する
- WBS 冒頭の `最終更新` がある場合は当日の日付に更新する
- 同じタスクについて詳細セクションに明示的な完了欄や完了メモがすでにある場合だけ、それも整合させる

## CHANGELOG の更新方針

新しい履歴はファイルの上側に入れる。`# CHANGELOG` の見出しがある場合は、その直下に最新エントリを入れる。

追記フォーマットは毎回この2行に統一する。

```text
[YYYY/MM/DD]短いタスク名（要約）
実装した内容の簡潔なまとめ
```

書き方:

- 1タスクにつき2行だけ書く
- 1行目は日付 + 短いタスク名
- 2行目は実装詳細の列挙ではなく、変更の中身が分かる自然文にする
- 既存ファイルに古いテンプレートや説明文が残っていても、新しい履歴はこの形式で先頭側に積む

良い例:

```text
[2026/04/18]R-02 Backspace段落統合
同一章内で段落先頭の Backspace 時に前段落へ統合されるよう実装した
```

避ける例:

```text
[2026/04/18]R-02
いろいろ修正
```

## 完了時の報告

更新したタスク ID、WBS の完了反映、CHANGELOG の追記内容を簡潔に報告する。
