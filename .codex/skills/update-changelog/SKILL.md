# Update Changelog

完了した作業を `CHANGELOG.md` に反映する。

このスキルのゴールは、後から「何が完了し、なぜ履歴に残ったか」を追える状態にすること。

## 成果物

- `CHANGELOG.md`: 先頭側に追加する2行の変更履歴

## 成功条件

- `CHANGELOG.md` の先頭側に今回の2行が追加されている
- 不要な大規模整形や無関係な変更が入っていない
- 完了扱いにした範囲と履歴の要約が一致している

## 制約

- WBS や他のタスク台帳は更新しない
- 関係ない過去ログの文体統一だけを目的に全面書き換えしない
- 新規タスクや残課題は、必要なら GitHub Issue への記録を提案する
- タスク ID、完了範囲、要約のいずれかが曖昧なときだけユーザーに確認する

## CHANGELOG の更新方針

新しい履歴はファイルの上側に入れる。`# CHANGELOG` の見出しがある場合は、その直下に最新エントリを入れる。

追記フォーマットは毎回この2行に統一する。

```text
[YYYY/MM/DD]短いタスク名（要約）
実装した内容の簡潔なまとめ
```

書き方:

- 1タスクにつき2行だけ書く
- 1行目は日付 + 短いタスク名
- 2行目は実装詳細の列挙ではなく、変更の中身が分かる自然文にする
- 既存ファイルに古いテンプレートや説明文が残っていても、新しい履歴はこの形式で先頭側に積む

良い例:

```text
[2026/05/07]文書内検索導線
現在開いている文書内のタイトル・章タイトル・段落本文を検索し、結果クリックで対象段落へ移動できるようにした。
```

## 完了時の報告

CHANGELOG の追記内容を簡潔に報告する。
