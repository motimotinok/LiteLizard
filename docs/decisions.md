## [2026-03-14] 分析機能の役割・製品構造を再定義

- **決定**: 分析ペインの役割を「読者の感想・評価」ではなく「書き手の意図が読者にどう届いたか（意図と解釈の擦り合わせ）」として定義する
- **理由**: 「面白かった」系の評価フィードバックは問題指摘マシンになりやすく、執筆体験を損なう。「この段落はこういう意図として読み取られる」という形式にすることで、書き手が自分で判断できる情報を渡せる
- **製品構造の整理**:
  - コア機能 = 書きやすさの追求（縦書き・横書き対応、執筆体験の洗練）
  - 分析 = 完成後レビューとして位置づけ（執筆中のリアルタイム分析ではない）
  - 分析機能はサブスク限定の付加価値として設計する
- **将来実装（MVPスコープ外）**:
  - 書き手の「意図宣言」フィールド（段落ごとに意図を記述）
  - 意図 vs 読者解釈のギャップ可視化
  - 感情アークのチャート表示（段落をまたいだ emotion タグの流れ）
- **却下した方向性**: 「問題点を指摘して書き直しを促すアシスタント」→ 書き手の主導権を損なうため採用しない
- **この決定が現在の実装に与える影響**: deepMeaning の文体方針（読者体感 + 書き手への気づきの混在）はこの定義と整合している

## [2026-03-20] .lzl 内部フォーマット: フロントマター + インラインマーカー方式を採用 (S-01)

- **決定**: .lzl ファイルを YAML フロントマター + HTML コメント形式のインラインマーカー（`<!--:: ch/p {id} ::-->`）で構造化する。フォーマットバージョン `lzl-v1`
- **仕様**: `docs/specs/lzl-format.md`
- **理由**:
  - ID と本文が物理的に隣接 → 部分破損時の被害が局所化される（破損耐性の最優先原則）
  - 1ファイルで構造が完結する（自己記述性）。外部メタファイルに依存しない
  - 空行 = 段落境界ルールにより、マーカーが消失しても段落境界を推定可能
  - フロントマターに章・段落カウントを冗長データとして持ち、パース後に破損検出が可能
  - 章タイトルはマーカー内に1行完結で格納（`## 見出し` との誤認を防ぐ）
- **却下した案**:
  - ヘッダー集約方式: 本文と構造の物理的距離が大きく、部分破損時に構造全体が影響を受ける
  - ブロック区切り方式: 冗長性は高いが構文が重く、パーサーの複雑化に見合わない
  - メタデータ分離方式（.lzl + .lzl.meta）: 2ファイル同期の破綻リスクが破損耐性と矛盾。メタファイル喪失で全構造消失
- **前提条件（この前提が崩れたら再検討）**:
  - Git 管理しない
  - 段落内改行を許容しない
  - 外部エディタでの編集を想定しない（閲覧のみ）

## [2026-03-27] 認証・セッションフロー設計 (S-05)

- **決定**:
  1. **ログイン方法**: OAuth のみ（メール+パスワード・マジックリンクは採用しない）
  2. **セッショントークン保存**: Electron `safeStorage` API を使用（macOS: Keychain、Windows: DPAPI、Linux: libsecret によるOS暗号化）
  3. **未ログイン時の挙動**: 分析パネルのみ変化。パネルを開くとログインボタン＋「ログインしてください」メッセージを表示。それ以外の機能（エディタ・ファイル操作）はログイン状態に関わらず動作する
- **理由**:
  - OAuth のみにすることでパスワード管理・ハッシュ保管の実装責任を排除できる
  - `safeStorage` は Electron 公式推奨の暗号化ストレージ。平文ファイルや localStorage より安全で、OS キーチェーン連携によりアプリ外からの読み取りを防止
  - 分析はサブスク限定機能（[2026-03-14] の製品構造決定と整合）なので、ゲートは分析パネルのみで十分
- **却下した案**:
  - メール+パスワード: パスワードハッシュ管理・リセットフロー実装コストが高い
  - マジックリンク: メール送信インフラが必要
  - localStorage へのトークン保存: XSS で窃取可能なため不採用
  - 全機能をログイン必須にする: エディタ単体の無料利用価値を損なうため不採用
- **仕様**: `docs/specs/auth-session.md`
- **影響範囲**: A-01〜A-05（認証・API 接続カテゴリ）がこの決定をベースに着手可能になる
- **未決定事項**: OAuth プロバイダー（Google / GitHub / 他）の選定は A-01 着手時に決定する

## [2026-03-28] 完全自動保存モデルを採用 (S-02)

- **決定**: 明示保存（Ctrl+S）を廃止し、完全自動保存に統一する。テキスト編集は 300ms デバウンスで即座に保存。DnD 並び替えは即確定だがテキスト編集より低優先度で保存キューに乗せる。dirty インジケータは表示しない
- **仕様**: `docs/specs/save-model.md`
- **理由**: ユーザーがタイピングと内容に集中したい。保存操作は面倒で意識したくないという明確な要望。デバウンス 300ms なら未保存を意識する時間窓が実質なく、dirty インジケータも不要になる
- **却下した案**:
  - 明示保存（Ctrl+S）: ユーザーが保存操作を意識する必要があり、執筆体験を損なう
  - 長めのデバウンス（2.5秒、現実装）: データ損失リスクが高く、ユーザーの「即座に保存されてほしい」要望と合わない

## [2026-03-28] 解析 API エンドポイント仕様 (S-06)

- **決定**: 段落単位の分析を基本とし、複数段落をまとめて1リクエストで送信。レスポンスは SSE ストリーミングで段落ごとに逐次返却。コンテキストは対象段落より前の最大10段落。結果はローカルのみ保存（`.litelizard/analysis/{documentId}_{generation}.json`）し、世代管理で構造変更時にファイルを一新する。認証は OAuth またはユーザー自前 API キー（後者はログイン不要）の2系統
- **仕様**: `docs/specs/analysis-api.md`
- **理由**: サーバーをステートレスにすることでシンプルさとプライバシーを確保。SSE により長時間の一括分析でもユーザーが逐次結果を確認できる。コンテキスト上限10段落はトークンコストと分析精度のバランス。「読者は前から順に読む」ため後方段落は参照しない（製品コンセプトとの整合）
- **却下した案**:
  - 段落ごと個別リクエスト並列方式: リクエスト数過多、コンテキスト共有不可
  - 一括レスポンス方式: 章・全体分析時の待ち時間が長い
  - サーバー側で結果保存: 複雑化、プライバシーリスク

## [2026-04-02] 章削除・段落統合エッジケース仕様 (S-07)

- **決定**: 章削除時は段落を前の章末尾に吸収。先頭章・最後の1章の削除時は「無題の章」を生成して段落を保持。Backspace による段落統合は同一章内のみ（章をまたがない）。空章は許容。確認ダイアログなし
- **仕様**: `docs/specs/chapter-paragraph-ops.md`
- **理由**: シンプルさを優先。章境界を Backspace で越えられると意図しない構造変更が起きやすい。確認ダイアログは将来の Undo/Redo（S-08）で代替する。空章を許容することで DnD 操作の自由度を確保
- **却下した案**:
  - 先頭章削除時に次の章へ吸収: 「前の章に吸収」ルールと一貫しない
  - 章をまたぐ Backspace 統合: 意図しない章境界の消失リスク
  - 空章の自動削除: ユーザーが意図的に空章を残すケースを阻害

## [2026-04-03] Undo/Redo 対象範囲と実装方針 (S-08)

- **決定**: 全操作（テキスト編集・章削除・段落統合/分割・DnD 並び替え）を Undo/Redo 対象とし、Zustand 統合 Command スタックで一元管理する
- **仕様**: `docs/specs/undo-redo.md`
- **理由**: Lexical HistoryPlugin（テキスト編集のみ）と構造操作（Zustand）の2系統が分裂すると Ctrl+Z の挙動が不整合になる。Lexical editorState はシリアライズ可能（`toJSON`/`parseEditorState`）なので、Zustand document スナップショットとセットで保存・復元する統合スタックに一本化できる
- **却下した案**:
  - Lexical HistoryPlugin を維持 + 構造操作を別スタック管理: Ctrl+Z の優先順位制御が複雑になり、テキスト編集と構造操作の間で Undo 順序が保証できない
- **テキスト編集の粒度**: Lexical デフォルトのグループ単位を踏襲（1打鍵ごとではなくグループ化された単位で Undo）
