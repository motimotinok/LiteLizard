# LiteLizard Product Map

最終更新: 2026-05-11

このファイルは、LiteLizard の全体像と今後伸ばしたい方向を LLM と人間が共有するための地図。作業台帳ではないため、細かい完了状態は持たない。実装可能な作業は `docs/tickets/`、完了履歴は `CHANGELOG.md`、確定した仕様は `docs/specs/` と `docs/decisions.md` に置く。

LLM が次に何を見るか迷った時は、まずこのファイルを読む。ここで方向と現在地を確認し、実行対象は `docs/tickets/`、完了済みの事実は `CHANGELOG.md` と `docs/tickets/done/`、詳細仕様は `docs/specs/` と `docs/decisions.md` へ辿る。

## プロダクトの核

- ローカルファーストの Electron アプリとして、`.lzl` 文書を安全に読み書きする。
- 小説・長文制作向けに、章・段落・分析カードの対応関係を保つ。
- 読者視点の分析を、外部 API またはローカル LLM で実行できるようにする。
- 編集体験は、構造操作、検索、表示調整、分析結果の参照を自然に往復できる状態に保つ。

## 現在の運用

- 実装キューは Ralph Loop チケット（`docs/tickets/*.md`）に一本化する。
- 完了済みチケットは `docs/tickets/done/` に残す。
- WBS は `docs/old/wbs.md` に退役済み資料として保存し、今後は更新しない。
- GitHub Issue は、将来構想・検討メモ・外部参照として必要な時だけ使う。
- `agent-ready` / `in-progress` による Issue 自律実行フローは退役済み。

## 現在地

その時点での開発の重心を 1〜2 文で書く。詳細は下の「進行中のテーマ（Now）」と「次に着手するテーマ（Next）」を見る。

- 直近の重心: 外部公開に向けて、作成文章を安全に外へ出す導線、初回/再起動時の保存先体験、分析実行前の安全確認、配布パッケージ生成を整える段階。
- 現在の状態: 公開準備の未完了 Ralph Loop チケットが作成済みで、次回以降は `docs/tickets/` 直下から順に実装できる。
- 直近で大きく整った領域は「成熟している領域」を見る。

## ロードマップ

実装の順序を「Now / Next / Later」の 3 段で持つ。各テーマには**前提（人間が判断するテーマ）**を 1 行添えて、依存関係が分かるようにする。チケットの細かい進捗はここでは持たない。

### Now（直近のチケット数本で扱う）

実装に落ちる手前まで来ているテーマ。Ralph Loop で順次チケット化していく対象。

- **公開準備の最小実装** ─ テキストエクスポート、前回ディレクトリ復元、分析実行前の見積もり確認、最小 packaging を整える。
  - 前提: まず Mac 向け Electron アプリの外部β配布を想定し、署名・notarization・自動更新は後続判断に分ける。

### Next（仕様判断後に着手）

方向は固まりつつあるが、設計判断や前提が片付いてから Now に上げるテーマ。

- **章解析・全体解析の実行体験** ─ 段落分析の集約に留めず、章/全体を独立した実行単位として整える。
  - 前提: 「段落分析の集約」と「別 LLM 呼び出し」のどちらにするかの判断。
- **Reading Agent を中核体験に育てる** ─ 読者プロファイル・プロンプト・分析実行を一連の流れにする。
  - 前提: Reading Agent の最初の完成形のスコープ（どこまでを v1 にするか）。
- **長文制作の迷子防止強化** ─ 検索、要約、履歴、差分確認の連携を強める。
  - 前提: いま不足しているのが横断検索なのか、章単位の俯瞰なのかの切り分け。

### Later（構想段階）

実装よりも先に方向の判断が必要、または実装は当面後回しにするテーマ。`#番号` は GitHub Issue を指す。

- **ローカル LLM 体験の本格化**（モデル管理・失敗復旧まで含めた扱いやすさ）─ #72 LiteLizard 管理のローカル LLM ランタイム。
- **拡張機能・カスタムワークフロー** ─ #73 ユーザー自作の拡張入口、#74 カスタムワークフローの自作ボタン。
- **Reading Agent のチューニング項目見直し** ─ #86 temperature 設定を将来削除するか。
- **クラウド連携の去就** ─ クラウドログインやクラウド解析を、将来拡張として残すだけでよいか。

## 成熟している領域

「Now でいま手を入れる対象ではない」領域。新しいテーマがここに依存することはあっても、地図上は背景として扱う。詳細は `CHANGELOG.md` と `docs/specs/` を見る。

- `.lzl` フォーマット、documentId、保存モデル、プロジェクト管理
- Electron IPC / preload bridge / file service
- ローカル分析基盤、API キー保存、外部 API / Ollama 接続
- 段落分析、分析履歴、stale 表示、全体解析件数表示
- 章・段落の基本構造操作と Undo/Redo
- UI リデザイン後の検索、Recent files、Tweaks、ローカルフォント、分析カード連動表示

## 人間が判断するテーマ

ロードマップの前提として未解決の判断。ここが解けると Next → Now の順に動く。

- Reading Agent の最初の完成形をどこまでにするか。（→ Next: Reading Agent の中核化）
- 章解析・全体解析を「段落分析の集約」に留めるか、別 LLM 呼び出しとして設計するか。（→ Next: 章解析・全体解析の実行体験）
- LiteLizard 管理のローカル LLM ランタイムを本当に持つか。（→ Later: ローカル LLM 体験の本格化）
- ユーザー自作拡張やカスタムワークフローを、製品の主要導線に入れるか。（→ Later: 拡張・ワークフロー）
- クラウドログインやクラウド解析を、将来拡張として残すだけでよいか。（→ Later: クラウド連携の去就）

## チケット化の基準

- 実装・テスト・検証まで 1 つのループで扱えるなら `docs/tickets/` に置く（= Now を消化する動作）。
- 仕様判断が先に必要なら、チケット化せず本ファイルの Next または判断テーマに残す。
- 思いつきは、すぐ実装するなら `docs/tickets/`、まだ曖昧なら本ファイル、外部参照や長期構想として残したいなら GitHub Issue に置く。
- 本ファイルから「やる」と決めたものは、Now に上げてから実装単位に分解して `docs/tickets/` に移す。本ファイル側では細かい進捗管理をしない。
- 完了後の履歴は `CHANGELOG.md` に残す。
- 設計判断として後から参照したいものは `docs/decisions.md` に残す。

## 関連スキル

- `update-product-map`: 思いつき・将来方向・判断テーマ・Now/Next/Later の入れ替えをこのファイルに整理する。
- `create-ralph-ticket`: Now に上げると決めた内容を `docs/tickets/` に切り出す。
- `update-changelog`: 完了した変更を `CHANGELOG.md` に記録する。
