## 方針・覚書

> WBS に載せない進め方の方針や検討事項をここに記録

- **S 系タスクの進め方**: 壁打ち形式でユーザーと会話 → 決まったら AI が specs/ と decisions.md に書き出す。着手の心理的ハードルを下げるため、各 S タスクに「壁打ちの始め方」を用意しておく
- 仕様 v003 の未策定箇所（.lzl フォーマット設計 §13 など）は仕様精緻化フェーズで詰める
- テスト整備は仕様が固まった領域から順次進める方針
- WBS 完成後、既存の未着手タスク（T26〜T28 等）は優先度を再評価して wbs.md に配置する
- **worktree 管理**: `isolation: "worktree"` に移行済み（2026-03-22）。work1〜work5 の手動 worktree は廃止。Claude Code が自動で worktree を作成・管理する
- **session-summary.md 廃止・TASKS.md への一本化** (2026-03-21): セッション引き継ぎ情報を TASKS.md に一本化。`docs/session-summary.md` は今後更新しない
- **開発フロー変更** (2026-03-21→03-22更新): Codex 並列配分を廃止、Claude 一気通貫方式に移行。固定ブランチ `claude/task` を廃止し、dev + `isolation: worktree` 方式に変更
- **APIキー管理**: S-09 完了により仕様 §9 を改訂し、クライアント側 API キー管理が正式方針に統一された
- **ローカル完結方針の合理性と留意点** (2026-03-29):
  - MVP段階ではサーバー運用コスト回避・プライバシー親和性・認証/課金の複雑さ排除の観点からローカル完結は合理的
  - **収益モデルが白紙に近くなる**: 「分析はサブスク限定」の前提が崩れる。ユーザーが自分のAPIキーを使うならLiteLizard側の課金ポイントがない。ただしMVP段階で検証すべきは「分析体験に価値があるか」であり課金モデルではないので、今は許容する。将来「この体験に金を払う価値がある」と検証できたら、マネージドサービス（LiteLizard側でAPIキー管理+サブスク課金）を追加する拡張パスは残る
  - **分析品質のコントロールが難しくなる**: ローカルLLM（Ollama）はモデルにより品質がバラバラ。外部APIキー方式でもモデル次第で結果が変わる。「意図と解釈の擦り合わせ」のコア体験品質がユーザー環境依存になる → 対策: 推奨モデル/推奨設定のドキュメント化、プロンプトテンプレートをLiteLizard側で提供してモデル差を吸収
  - **拡張方向**: ローカル完結→マネージドへの拡張は可能だが逆は難しい。まずローカルから始めるのは順序として正しい
- **ExplorerPane**: 空白クリック時にファイル選択状態（グレー背景）が解除されない問題あり。要検討

### WBS に載っていない検討事項

**製品としての刺さりの再検討**
- 「意図と解釈の擦り合わせ」（decisions.md 2026-03-14）はコンセプトとして決めたが、実際のユーザー体験としてどう感じるかは未検証。モックで数段落分の分析結果を自分で使ってみて、「これ欲しい」と思えるか確認するフェーズが必要
- 分析結果の表示形式（現状のカード型）が最適かは疑問。段落の横に出るのか、別ペインか、ポップオーバーか — 形式次第で「便利」と「邪魔」が分かれる。E-03（IPC接続）前に一度プロトタイプで体感を確かめたい
- コア機能（書きやすさ）が分析なしでも魅力的か？ 現状のエディタ体験はミクロ/マクロ切り替え + DnD だが、これだけで既存エディタ（VS Code, Scrivener, 一太郎）から乗り換える理由になるか

**ターゲットユーザーの具体化**
- 「小説家」は幅が広い。Web小説の量産型か、文芸誌投稿の推敲型か、趣味の同人勢か。分析機能の価値が最も高いのはどの層か
- 縦書き対応は日本語文芸ユーザーへの訴求力が高いが、実装コストも高い。MVP に含めるかの判断が未確定

**収益モデルの具体化**
- 「分析はサブスク限定」は決まっているが、価格帯・課金単位（月額/文字数/回数）は未検討
- エディタ無料 + 分析サブスクの場合、無料エディタ単体の魅力が十分でないと導線が機能しない

**初回体験（オンボーディング）**
- 初めてアプリを開いた人が「何をするアプリか」を理解して最初の文書を作るまでの導線が未設計
- 空の状態（Empty State）のUI、サンプル文書の同梱、チュートリアルの要否

**ベータテスト計画**
- いつ、誰に、どう使ってもらうか。GitHub Pages デモは公開済みだが、デスクトップ版をどの段階で外部に渡すか
- フィードバック収集の仕組み（アプリ内フォーム？ Discord？ Google Form？）
