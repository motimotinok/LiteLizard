# プロダクトレビュー: LiteLizard (2026-03-20)

> プロダクト全体の方向性・市場価値・持続可能性について、仕様v003・decisions.md・実装状況を踏まえたレビュー。
> 特に「1〜2年後にAIで開発がコモディティ化する世界で持続的な価値を持てるか」を重視。

---

## 本能分析

### 現状の訴求マップ

| 本能 | 強度 | 根拠 |
|------|------|------|
| **安らぐ** | 弱 | 自動保存・リビジョン競合チェックはあるが、「安心感」を能動的に伝えるUXではない |
| **進める** | 弱 | 文字数表示のみ。執筆の進捗感を与える仕組みが薄い |
| **決する** | 中 | ミクロ/マクロ切り替え、モード選択、縦書き/横書き（将来）。「自分のやり方で書ける」感はある |
| **有する** | 中 | .lzlという独自形式、プロジェクト管理、ファイルが自分のローカルにある所有感 |
| **属する** | なし | 完全に個人ツール。コミュニティ・共有の要素がゼロ |
| **高める** | **強** | コア。「意図と読者解釈のギャップ」を可視化し、書き手の上達を支援する |
| **伝える** | なし | エクスポート機能が仕様にない。書いたものを外に出す導線がない |
| **物語る** | 中 | マクロ視点での章構成俯瞰、段落の並び替え。「自分の物語を構造として見る」体験 |

### 深掘りポイント

**「高める」が最大の武器だが、まだ浅い。**

現状の分析機能は「読者がこう読み取る」というフィードバックを返すだけ。これだと：
- 一回見て「ふーん」で終わるリスクがある
- **上達の実感**がない。「先月の自分より良くなった」がわからない

decisions.mdの「意図宣言フィールド」「感情アークのチャート」は正しい方向。特に感情アークは「自分の文章の流れを視覚的に見る」体験になり、**物語る**本能も同時に強化する。

### 拡張の可能性

**「伝える」が完全に欠落している。致命的。**

書き手は最終的に「誰かに読んでもらいたい」。LiteLizardで書いたものを外に出す導線がない：
- .lzl → .txt / .md / .docx / .pdf エクスポート
- コピペ用のプレーンテキスト出力
- 最低限これがないと「書いて終わり」のツールになる

**「安らぐ」をもっと意識的に設計すべき。**

執筆ツールにおける「安らぐ」は：
- 「書いたものが消えない」安心感（自動保存の可視化）
- 「分析結果に怯えなくていい」安心感（問題指摘マシンにならない設計 — decisions.mdで正しく判断済み）
- 「いつでも戻れる」安心感（Undo/Redoの充実）

## 失敗シナリオ

### 最も警戒すべきシナリオ

**1. 「で、これWordやNotionの代わりに使う理由は？」問題**
- 影響: 致命的 / 発生確率: **高**

ターゲットユーザーが小説・エッセイを書く人だとして、彼らは今すでに何かで書いている。LiteLizardに乗り換えるには移行コストを上回る「ないと困る」体験が必要。

現状の分析機能は「あると嬉しい」止まりのリスクが高い。なぜなら：
- 分析はサブスク限定の付加価値として設計されている
- つまりコア体験は「エディタとして書きやすいこと」
- しかしエディタとしての差別化が「ミクロ/マクロ視点切り替え」「段落DnD」で、これは便利だが乗り換え動機としては弱い

**対策案:**
- コアの「書く体験」自体に、他では得られない何かが必要
- 縦書き巻物UIは実はここが強い。**日本語の縦書きをまともに扱えるモダンなエディタがほぼ存在しない**。これだけで一定のニッチを取れる可能性がある
- MVP段階で縦書きを後回しにしているのは合理的だが、プロダクトの差別化としては最優先で検討すべき

---

**2. 「1〜2年後にAIがその場でこれを作れる」問題**
- 影響: 致命的 / 発生確率: **中〜高**

ChatGPTやClaudeに「書いた文章を段落ごとに分析して、読者がどう読み取るか教えて」と言えば、今でもそれなりの結果が返ってくる。1〜2年後には：
- AIに文章を貼って「分析して」で済む
- AIが個人専用の執筆分析ツールを即座に生成する

LiteLizardの分析機能はAIのラッパーである以上、ラッパーの価値は薄れていく。

**対策案:**
- **分析機能を売りにしない**方向への転換を検討
- 代わりに「書く体験そのもの」——巻物UI、段落構造の可視化、章の俯瞰——をコアにする
- 分析はあくまで付随機能として、エディタ体験に統合されていること自体が価値（別ツールに貼り直す手間がない）
- もしくは、分析の蓄積・履歴（「この段落は3回書き直して、こう変化した」）のような**時間軸を持つデータ**はAIの一発分析では代替できない

---

**3. 「サーバーAPIへの依存 = サービス終了リスク」問題**
- 影響: 致命的 / 発生確率: 中

分析機能がサーバーAPI必須・サブスク限定。これは：
- サービスが続かなくなったとき、分析機能が全部死ぬ
- ユーザーにとって「このツールに依存して大丈夫か」という不安になる
- 特に作品を書く人にとって、ツールのライフサイクルは重大な関心事

**対策案:**
- .lzlがローカルファイルで読み取り可能な設計は正しい（エディタ自体はオフラインで動く）
- 将来的にローカルLLM対応（ollama等）のオプションを示唆するだけでも安心感になる

### 注意すべきシナリオ

**4. MVP が「中途半端なエディタ + 中途半端な分析」になる**
- 影響: 痛い / 発生確率: 中

MVPスコープが広い。エディタ（ミクロ・マクロ・DnD・章CRUD・Undo）+ 分析 + エクスプローラー + 認証。全部「動くけど磨かれていない」状態でリリースすると、どの軸でも既存ツールに負ける。

**対策:** 「これだけは最高に気持ちいい」という体験を1つ選んで磨く。全部70点より、1つだけ120点のほうが印象に残る。

---

**5. .lzl 独自形式のロックイン感**
- 影響: 痛い / 発生確率: 中

ユーザーが「このフォーマットで書いて大丈夫？」と不安になる。エクスポートがないと致命的。逆に、.lzlのメリット（段落ID・章構造の永続化）がユーザーに伝わらないと「なぜMarkdownじゃダメなの？」になる。

### 許容できるリスク

- **縦書き未実装でのMVPリリース** — 横書きだけでも機能は成立する。ただし差別化の機会損失は認識すべき
- **章サマリー分析の後回し** — 段落分析だけでもコア体験は伝わる

---

## 総合所見

**最大の強み:** 「意図と読者解釈のギャップを可視化する」というコンセプトは独自性がある。decisions.mdの方向転換（評価フィードバック → 意図の擦り合わせ）は正しい判断。

**最大のリスク:** 分析機能はAIのラッパーとして価値が急速に減衰する可能性が高い。一方、**書く体験そのもの**——縦書き巻物UI、段落という構造単位で文章を扱うUX、ミクロ/マクロの視点切り替え——は「その場で生成」では代替しにくい。**コアを「分析」から「書く体験」にシフトすべきかどうか**が、今最も重要な戦略判断。

---

## 戦略的な問い（未決定）

- 「生産性ツール」として出すと1〜2年で価値を失う可能性が高い
- 「書くことを豊かにする体験」——巻物のように文章を広げていく感触、自分の文章の構造を俯瞰する体験、意図と解釈のギャップに気づく瞬間——効率ではなく**質感**の方向に振れるかどうかが分岐点
- 人間のスローな営みを支援するもの、または作品としてのソフトウェア——どちらの方向に寄せるか
