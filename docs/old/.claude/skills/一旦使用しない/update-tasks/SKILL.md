# Update Tasks — タスクダッシュボード更新スキル

ユーザーから聞いたタスクや状況変化を `PROJECTMEMORY/TASKS.md` に反映するスキル。

このスキルが存在する理由: セッション開始時に TASKS.md を読むだけで「今何をすべきか」が迷いなく分かる状態を維持するため。タスクの粒度・優先度・背景情報を適切に管理し、作業開始までの認知コストをゼロに近づける。

---

## TASKS.md のフォーマット

TASKS.md は以下のセクション構成を維持する:

```markdown
# LiteLizard ダッシュボード

## NEXT: {タスク名}（{優先度}）

> セッション開始時にここだけ読めば作業に入れる状態にする。

- **背景**: なぜこのタスクが必要か。やらないとどうなるか
- **アクション**:
  1. 具体的な手順1
  2. 具体的な手順2
- **対象ファイル**: `path/to/file.ts`（分かっている場合のみ）
- **完了条件**: 何ができたら終わりか

---

## タスクキュー（優先度順）

### {タスクID}: {タスク名}（{優先度}）
- **背景**: なぜこのタスクが必要か
- **アクション**:
  1. 手順
- **対象ファイル**: （分かっている場合のみ）
- **完了条件**: 何ができたら終わりか

### {タスクID}: {タスク名}（{優先度}）
...

---

## 方針・覚書

> WBS に載せない進め方の方針や検討事項をここに記録

---

## 完了メモ

> 完了したタスクの簡易記録。10件を超えたら古い順に ARCHIVE.md へ移動

- {タスク名}（{完了日}）— 一行サマリ

---

## アイデアボックス

> 今は着手しないが将来やりたいこと
```

## 更新の手順

### Step 0: 現状を読む

1. `PROJECTMEMORY/TASKS.md` を読む
2. 必要に応じて `docs/wbs.md` を参照する（WBS 由来タスクの詳細確認）

### Step 1: 何をすべきか判断する

ユーザーの発話から、以下のどの操作が必要か判断する:

| ユーザーの意図 | 操作 |
|---------------|------|
| 新しいタスクを伝えてきた | タスクキューに追加 |
| タスクが完了した | NEXT またはキューから削除 → 完了メモへ |
| 優先度を変えたい | 並び替え、必要なら NEXT を差し替え |
| 状況が変わった | 該当タスクの記述を更新 |
| タスクを取り消したい | キューから削除（完了メモには入れない） |

### Step 2: タスクの粒度と内容を決める

**タスク記述の原則:**

1. **背景セクション** — そのタスクが必要な理由を書く。「状況」ではなく「動機」にフォーカスする
   - 良い例: 「.lzl パーサーに P1 バグがあり、破損した章マーカーが素通りする。これが直らないと E-03 以降の IPC 接続タスクに進めない」
   - 避ける例: 「PR #53 でレビュー指摘があった」（状況は変わる。なぜ直す必要があるかが重要）

2. **アクション** — 読んだ人が迷わず手を動かせる粒度で書く。ファイル名や関数名を含める

3. **完了条件** — 曖昧な表現を避け、検証可能な条件にする

### Step 3: 優先度を判断して並べる

**優先度の判断基準:**

| 優先度 | 基準 |
|--------|------|
| P0 | ブロッカー。これがないと他のタスクが進まない |
| P1 | 製品公開に必須、またはユーザーが急ぎと言った |
| P2 | あると良い。公開後でも可 |
| P3 | 将来検討 |

- WBS 由来のタスクは WBS の優先度をベースにする
- ユーザーから聞いたタスクは、文脈から優先度を推測する。迷ったら聞く
- 依存関係があるタスクは、依存先が未完了でもキューに載せてよい（背景に依存関係を明記する）
- **NEXT にはキューの最上位タスクを置く**。ユーザーが明示的に指定した場合はそれに従う

### Step 4: タスクID の採番

- WBS に載っているタスクはそのID（S-01, E-02 等）をそのまま使う
- WBS に載っていないタスク（ユーザーから聴取したもの、ツール改善等）は `U-{連番}` を使う
  - 例: `U-01`, `U-02`
  - 既存の U-xx を確認して重複しない番号を振る

### Step 5: TASKS.md を更新する

- フォーマットのセクション構成を維持する
- タスクキューは優先度順（P0 → P1 → P2 → P3）、同一優先度内はブロッカー性が高い順
- 完了メモが10件を超えたら、古い順に `PROJECTMEMORY/ARCHIVE.md` へ移動する

### Step 6: ARCHIVE.md への退避（必要な場合のみ）

完了メモが10件を超えている場合:

1. `PROJECTMEMORY/ARCHIVE.md` を読む
2. 古い完了メモを日付セクション付きで ARCHIVE.md の先頭（既存アーカイブの上）に追加
3. TASKS.md の完了メモから移動した分を削除

## 注意事項

- **WBS は触らない**: このスキルは TASKS.md のみを更新する。`docs/wbs.md` は別途手動で更新する
- **方針・覚書セクション**: タスクではない「進め方の方針」「検討事項」はこのセクションに置く。タスクキューと混ぜない
- **アイデアボックス**: 優先度がつけられないもの、「いつかやりたい」レベルのものはここに置く
- **懸念・リスクセクション**: 廃止。リスクはタスクの背景に含めるか、方針・覚書に記載する
