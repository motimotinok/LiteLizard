# Spec Writer — 仕様文書化スキル

仕様策定の結論を、実装判断に使える形で記録する。

このスキルのゴールは、「何を決めたか」と「なぜそうしたか」を分けて残し、後続の実装者が既存仕様を壊さず判断できる状態にすること。

## 成果物

必ず3箇所を同期する。

- `docs/specs/*.md`: 現行仕様、制約、観測可能な振る舞い、検証観点
- `docs/decisions.md`: 決定理由、却下した案、判断の背景
- `docs/specs/README.md`: 仕様ファイルのインデックスとステータス

## 成功条件

- specs は、実装手順を固定する文書ではなく、実装判断に必要な契約を示している
- decisions.md は、仕様の詳細ではなく「なぜその判断にしたか」を短く残している
- README のインデックスから対象仕様へ辿れる
- 関連タスク ID、仕様ステータス、決定日が分かる
- 既存ファイルを更新する場合、関係ない章や過去の決定を大きく書き換えていない

## 制約

- 仕様の詳細と決定理由を同じ場所に重複して書かない
- 型、API、保存形式、IPC など守るべき契約がある場合だけ、具体的なコード例を使う
- 実装者が選べる経路まで細かく固定しない
- 未確定事項は確定したように書かず、`draft` や未決定事項として残す
- 情報が不足している場合は、会話・WBS・既存specs・decisionsを確認し、それでも足りなければユーザーに確認する

## 書く内容

specs には次を中心に書く。

- 背景ではなく、現在有効な仕様
- 対象機能、入力、出力、状態遷移、保存形式、UI挙動などの契約
- 守るべき制約と非ゴール
- 実装後に確認すべき検証観点
- 関連タスク ID と decisions.md へのリンク

decisions.md には次を中心に書く。

- 決定の短い要約
- なぜその判断にしたか
- 却下した案と理由
- specs へのリンク

## 固定フォーマット

`docs/specs/{topic}.md`:

```markdown
# {仕様タイトル}

関連タスク: {タスクID}
ステータス: draft / stable / deprecated
決定経緯: `docs/decisions.md` [{日付}] {タスクID}

---

## 1. 概要

## 2. 現行仕様

## 3. 制約・非ゴール

## 4. 検証観点
```

`docs/decisions.md`:

```markdown
## [{日付}] {決定タイトル} ({タスクID})

- **決定**: {1〜2行の要約}
- **仕様**: `docs/specs/{topic}.md`
- **理由**: {なぜこの判断をしたか}
- **却下した案**: {必要な場合だけ}
```

`docs/specs/README.md`:

```markdown
| [{topic}.md]({topic}.md) | {トピック説明} | {タスクID} | {ステータス} |
```

## 完了時の報告

変更した3ファイルを明示し、仕様・決定理由・インデックスを同期したことを簡潔に報告する。
