# .lzl ファイルフォーマット仕様 (lzl-v1)

最終更新: 2026-03-20

---

## 1. 概要

`.lzl` は LiteLizard 専用のテキストベース文書フォーマットである。

### 設計原則

| 原則 | 内容 |
|------|------|
| **破損耐性** | ID と本文が物理的に隣接し、部分破損の影響を局所化する |
| **自己記述性** | 1ファイルで構造が完結する。外部メタファイルに依存しない |
| **閲覧可能性** | 外部エディタで開いた場合、本文テキストが読み取れる |
| **修復可能性** | メタデータが部分的に壊れても、残存情報から構造を再構築できる |

### 前提条件

- Git 管理は想定しない
- 外部エディタでの編集は想定しない（閲覧のみ）
- 段落内改行はない（空行 = 段落境界）
- エクスポート（プレーンテキスト等）は別レイヤーで対応する

---

## 2. ファイル構造

`.lzl` ファイルは以下の3要素で構成される：

```
┌─────────────────────────┐
│ YAML フロントマター      │  ← 文書メタデータ
├─────────────────────────┤
│ 章マーカー              │  ← 章の開始
│ 段落マーカー + 本文     │  ← 段落（繰り返し）
│ 段落マーカー + 本文     │
│ ...                     │
│ 章マーカー              │  ← 次の章
│ 段落マーカー + 本文     │
│ ...                     │
└─────────────────────────┘
```

---

## 3. フロントマター

ファイル先頭に YAML フロントマターを配置する。`---` で囲む。

```yaml
---
documentId: d_a8f3k2m9x1
format: lzl-v1
title: 吾輩は猫である
chapters: 2
paragraphs: 5
created: 2026-03-20T00:00:00Z
updated: 2026-03-20T12:00:00Z
---
```

### フィールド定義

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `documentId` | string | ✅ | 文書の一意識別子。`d_[a-z0-9]{10}` |
| `format` | string | ✅ | フォーマットバージョン。現行は `lzl-v1` |
| `title` | string | ✅ | 文書タイトル |
| `chapters` | number | ✅ | 章の数（検証用冗長データ） |
| `paragraphs` | number | ✅ | 段落の数（検証用冗長データ） |
| `created` | ISO8601 | ✅ | 作成日時 |
| `updated` | ISO8601 | ✅ | 最終更新日時 |

### 冗長性の役割

`chapters` と `paragraphs` はパース後の実数と照合し、不一致があれば破損を検出するために使う。これらの値自体が壊れていても、本文のマーカーから正しい構造を再構築できる。

---

## 4. マーカー構文

本文中に HTML コメント形式のマーカーを埋め込み、章・段落の構造を表現する。

### 4.1 章マーカー

```
<!--:: ch {chapterId} | {title} ::-->
```

- `chapterId`: `c_[a-z0-9]{10}`
- `title`: 章タイトル（`|` の後、`::-->` の前）
- 章マーカーの直後は必ず空行を置く

**例:**
```
<!--:: ch c_x1y2z3w4a5 | 第一章 ::-->

```

### 4.2 段落マーカー

```
<!--:: p {paragraphId} ::-->
```

- `paragraphId`: `p_[a-z0-9]{10}`
- マーカーの直後の行から次の空行までが段落テキスト

**例:**
```
<!--:: p p_abc123def4 ::-->
吾輩は猫である。名前はまだ無い。

```

### 4.3 正規表現

マーカーの抽出に使用する正規表現：

```
章マーカー:   /^<!--::\s*ch\s+(c_[a-z0-9]{10})\s*\|\s*(.+?)\s*::-->$/
段落マーカー: /^<!--::\s*p\s+(p_[a-z0-9]{10})\s*::-->$/
```

### 4.4 なぜ `<!--:: ... ::-->` か

| 設計判断 | 理由 |
|----------|------|
| HTML コメント形式 | 外部エディタで開いた時にマーカーが「コメント」として認識される |
| `::` デリミタ | 通常の HTML コメント `<!-- ... -->` との誤認を防ぐ |
| 1行完結 | 複数行マーカーは部分破損のリスクが高い |

---

## 5. 段落区切りルール

- 段落マーカーの直後の行〜次の空行の直前までが段落テキスト
- **空行（連続する `\n\n`）が段落の終了を示す**
- 段落内改行は存在しない（1段落 = 1行のテキスト）
- マーカーが消失しても、空行で段落境界を推定できる

```
<!--:: p p_abc123def4 ::-->
吾輩は猫である。名前はまだ無い。
                                    ← 空行 = 段落終了
<!--:: p p_def456ghi7 ::-->
どこで生れたかとんと見当がつかぬ。
                                    ← 空行 = 段落終了
```

## 6. 構造ルール

### 6.1 全体構造

1. ファイルは YAML フロントマターで始まる
2. フロントマターの後、最初の要素は必ず章マーカー
3. 章マーカーの後に、その章に属する段落が続く
4. 段落は必ず直前の章に属する
5. 章・段落の順序はファイル内の出現順で決定する（`order` フィールドは不要）

### 6.2 最小構造

文書は最低1章・1段落を持つ。空の新規文書は以下の形式：

```yaml
---
documentId: d_xxxxxxxxxx
format: lzl-v1
title: 無題
chapters: 1
paragraphs: 1
created: 2026-03-20T00:00:00Z
updated: 2026-03-20T00:00:00Z
---

<!--:: ch c_xxxxxxxxxx | 第1章 ::-->

<!--:: p p_xxxxxxxxxx ::-->

```

最後の段落は空テキスト（マーカーの直後が空行）。

### 6.3 章と段落の帰属

```
<!--:: ch c_aaa | 第一章 ::-->

<!--:: p p_001 ::-->        ← c_aaa に属する
段落テキスト1

<!--:: p p_002 ::-->        ← c_aaa に属する
段落テキスト2

<!--:: ch c_bbb | 第二章 ::-->

<!--:: p p_003 ::-->        ← c_bbb に属する
段落テキスト3
```

段落の `chapterId` はファイルに明示的に書かない。パース時に「直前の章マーカー」から決定する。

## 7. ID 仕様

### 7.1 ID フォーマット

| 種類 | プレフィックス | パターン | 例 |
|------|---------------|----------|-----|
| 文書 | `d_` | `d_[a-z0-9]{10}` | `d_a8f3k2m9x1` |
| 章 | `c_` | `c_[a-z0-9]{10}` | `c_x1y2z3w4a5` |
| 段落 | `p_` | `p_[a-z0-9]{10}` | `p_abc123def4` |

### 7.2 ID の永続性

- `paragraphId` はテキスト編集・並び替えで変わらない
- `chapterId` は章の追加・削除・並び替えで変わらない
- `documentId` はファイルのリネームで変わらない
- ID が変わるのは: 破損時の自動修復、ファイルコピー時の重複解消のみ

### 7.3 ID 生成アルゴリズム

```typescript
function generateId(prefix: 'd' | 'c' | 'p'): string {
  return `${prefix}_${randomAlphanumeric(10)}`;
}
```

- `randomAlphanumeric(10)`: `[a-z0-9]` から10文字のランダム文字列
- 衝突確率: 36^10 ≈ 3.6 × 10^15（実用上無視できる）
- 全レイヤーで統一する（現実装の文字数不一致を修正）

## 8. パース仕様

### 8.1 パースフロー

```
1. ファイル全文を読み込む
2. フロントマター（--- 〜 ---）を YAML パースする
3. 本文を1行ずつ走査する:
   a. 章マーカーにマッチ → 新しい章を開始
   b. 段落マーカーにマッチ → 新しい段落を開始
   c. 空行 → 現在の段落テキストを確定
   d. その他 → 現在の段落テキストに追記
4. 検証フェーズ（§9）を実行
```

### 8.2 パース結果の型

```typescript
interface ParsedLzlDocument {
  frontmatter: {
    documentId: string;
    format: string;
    title: string;
    chapters: number;
    paragraphs: number;
    created: string;
    updated: string;
  };
  chapters: Array<{
    id: string;
    title: string;
  }>;
  paragraphs: Array<{
    id: string;
    chapterId: string;  // パース時に直前の章から決定
    text: string;
  }>;
}
```

## 9. 破損検出と自動修復

### 9.1 検証チェックリスト

パース後に以下を検証する：

| # | チェック | 検出方法 | 修復アクション |
|---|---------|----------|---------------|
| 1 | フロントマター欠損 | YAML パース失敗 | 本文マーカーから構造再構築。`documentId` を新規生成 |
| 2 | 章カウント不一致 | `frontmatter.chapters ≠ 章マーカー数` | 実数を正とする。フロントマターのカウントを更新 |
| 3 | 段落カウント不一致 | `frontmatter.paragraphs ≠ 段落マーカー数` | 実数を正とする。フロントマターのカウントを更新 |
| 4 | 章マーカーなし | 本文に章マーカーが0個 | デフォルト章を生成し、全段落を帰属させる |
| 5 | マーカーなしテキスト | 章マーカー後に段落マーカーなしのテキストブロック | 新しい `paragraphId` を生成して割り当てる |
| 6 | ID 重複（同一ファイル内） | 同じ ID が2回以上出現 | 2つ目以降に新 ID を振る |
| 7 | `documentId` 重複（プロジェクト内） | 同じ `documentId` を持つ別ファイルが存在 | 後から開いたファイルに新 `documentId` を振る（S-03 で詳細決定） |
| 8 | 段落が0個の章 | 章マーカー直後に次の章マーカーまたは EOF | 空テキストの段落を1つ生成 |

### 9.2 修復の原則

1. **本文テキストを最優先で保全する** — メタデータは再生成できるが、テキストは失ったら終わり
2. **UI を止めない** — 修復はバックグラウンドで実行し、ユーザーの編集を妨げない
3. **修復後に自動保存する** — 修復結果をディスクに書き戻し、次回オープン時は正常状態にする
4. **ログを残す** — MVP ではコンソール出力。将来的に UI 通知を追加

## 10. シリアライズ仕様

### 10.1 書き込みフロー

```
1. フロントマターを生成（章・段落カウントを実数で更新）
2. 各章について:
   a. 章マーカーを出力
   b. 空行を出力
   c. 章に属する各段落について:
      i.  段落マーカーを出力
      ii. 段落テキストを出力
      iii. 空行を出力
3. 一時ファイルに書き込み → rename でアトミック保存
```

### 10.2 アトミック書き込み

保存中クラッシュによる部分書き込みを防ぐため、以下の手順で保存する：

```
write  → /path/to/document.lzl.tmp
fsync  → ディスクに確実に書き出し
rename → /path/to/document.lzl
```

`rename` は OS レベルでアトミックな操作であり、`.lzl` ファイルが不完全な状態になることを防ぐ。

## 11. 完全な例

```yaml
---
documentId: d_a8f3k2m9x1
format: lzl-v1
title: 吾輩は猫である
chapters: 2
paragraphs: 5
created: 2026-03-20T00:00:00Z
updated: 2026-03-20T12:34:56Z
---

<!--:: ch c_x1y2z3w4a5 | 第一章 ::-->

<!--:: p p_abc123def4 ::-->
吾輩は猫である。名前はまだ無い。

<!--:: p p_def456ghi7 ::-->
どこで生れたかとんと見当がつかぬ。

<!--:: p p_jkl789mno0 ::-->
何でも薄暗いじめじめした所でニャーニャー泣いていた事だけは記憶している。

<!--:: ch c_m4n5o6p7q8 | 第二章 ::-->

<!--:: p p_pqr012stu3 ::-->
吾輩はここで始めて人間というものを見た。

<!--:: p p_vwx345yz67 ::-->
しかもあとで聞くとそれは書生という人間中で一番獰悪な種族であったそうだ。
```

## 付録: フォーマット変更時の移行

`format` フィールドでバージョンを管理する。将来 `lzl-v2` を導入する場合：

1. パーサーは `format` を読み、対応するパースロジックを選択する
2. `lzl-v1` → `lzl-v2` の移行関数を用意する
3. ファイルオープン時に自動変換し、保存時に新バージョンで書き出す
