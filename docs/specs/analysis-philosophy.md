ステータス: stable

## 1. 概要

LiteLizard の分析機能は、文章に対する唯一の正しい評価を返す機能ではない。ユーザーが定義した人格、立場、目的、評価基準を持つ Reading Agent を通して、文章がその観点からどう見えるかを観測するための環境である。

Reading Agent の内容はユーザーが自由に定義できる。一般読者、特定の人格を持つ読者、編集者、批評家、構成分析者、ファクトチェッカー、心理分析者、全肯定の伴走者などをシステム側で分類・分離・禁止しない。

アプリの責務は、評価内容を決めることではなく、次のレールを提供することである。

- Agent の作成、保存、選択、実行
- 分析対象と参照文脈の制御
- その回だけの追加指示
- 構造化出力の強制と検証
- 結果の表示、保存、復旧
- 実行前に送信範囲と使用量を把握できる透明性

文章を書く主体と最終判断者は人間である。AI は分析結果を返すだけの存在であり、本文の自動変更、ファイル操作、外部ツール実行などを行わない。

## 2. 現行仕様

### 2.1 中核原則

#### 2.1.1 評価基準はユーザーが所有する

LiteLizard は、一般的な「良い文章」の基準を Reading Agent へ暗黙に追加しない。

評価基準を与えずに良し悪しをAIへ丸投げすると、モデルが独自の暗黙基準を採用し、文章が目指している方向から遠ざかる可能性がある。したがって、何を評価し、何を評価しないかは Agent のプロンプトで定義する。

同じ文章でも、Agent の目的に応じて次のような結果を許容する。

- 読者にどう届くかという疑似体験
- 修正点や削除候補の指摘
- 書き手の意図と伝わり方のずれ
- 機能している部分、強み、残すべき構造の特定
- 誤記、重複、論理、構成など特定基準による評価
- 書き手の出力を止めないための積極的な称賛や伴走

称賛、批判、助言、修正案、書き換え例を一律に許可・禁止しない。すべて Agent の目的とプロンプトに従う。

#### 2.1.2 読みは特定の視点による観測である

分析結果は客観的な正解としてではなく、選択された Agent の視点から得た観測として扱う。

人格や読者像を与えられた Agent は、その人格になりきり、「私はここで不安を感じた」のような主観的な一人称の応答を返してよい。編集者や分析者として定義された Agent は、その役割に適した文体と評価軸を使ってよい。

システム側から「断定を避ける」「事実と推測を必ず区別する」「作者の心理を分析しない」といった一般的な内容制約を追加しない。心理分析やファクトチェックなど、それらの制約と衝突する用途もユーザーが選べるためである。

#### 2.1.3 分析はユーザーが明示的に呼び出す

分析は完成後だけに限定せず、執筆途中でも利用できる。ただし、アプリが完成度や執筆段階を推測して自動実行しない。

- リアルタイムで自動分析しない
- 本文変更だけを理由に自動再分析しない
- ユーザーの明示操作によってのみ実行する
- 分析結果を本文へ自動反映しない

#### 2.1.4 書き手が主導権を持つ

AI の応答は判断材料であり、採用すべき命令ではない。LiteLizard は、分析結果から本文を自動で書き換えることを分析機能の既定動作にしない。

将来、書き換え補助を追加する場合も、分析とは別の明示操作として扱い、人間が対象と変更内容を確認して適用する設計が必要である。

### 2.2 プロンプトの階層

分析実行時の入力は、次の順序で構成する。

1. LiteLizard の最低限の実行契約
2. Reading Agent のプロンプト
3. Agent に設定された文脈ポリシー
4. 今回だけの追加指示
5. 分析対象と参照文脈

今回だけの追加指示は Reading Agent の一時的な上書きとして扱い、内容が競合する場合は追加指示を優先する。ただし、分析対象に対する回答を返すことと、アプリが読み取れる構造化データを返すことは上書きできない。

構造化出力は provider の JSON schema 等を使ってプログラム側で強制する。出力形式を自然言語のプロンプトとして重ね、Agent の指示領域を汚染しない。

LiteLizard がすべての Agent に共通で要求する内容上の契約は、次の2点だけとする。

- 選択された分析対象に対する回答を返す
- アプリが読み取り、検証できる構造で返す

それ以外の評価基準、文体、立場、不確実性の扱い、助言の有無は Agent のプロンプトに委ねる。

### 2.3 その回だけの追加指示

ユーザーは、保存済みの Agent を編集せず、分析実行時だけ有効な追加指示を与えられる。

用途の例:

- 今回だけ特定の観点を加える
- 通常は助言しない読者Agentへ、今回だけ修正案を求める
- 目指している方向性や意図を一時的に伝える
- 特定の読者層、媒体、目的を補足する

追加指示は Agent の恒久的なプロンプトへ自動統合しない。履歴には追加指示の本文やハッシュを複製保存せず、追加指示が存在したかどうかだけを記録する。

### 2.4 分析対象と粒度

マクロ／ミクロ／超ミクロは分析内容を規定する分類ではなく、結果を紐づける分析対象の粒度である。

| UI上の視点 | 対象スコープ | 主な対象単位 |
|---|---|---|
| 超ミクロ | `sentence` | 一文 |
| ミクロ | `paragraph` | 段落 |
| マクロ | `chapter` | 章 |
| 文書全体 | `document` | 文書全体 |

粒度ごとの評価内容、出力項目、助言の有無をシステム側で固定しない。同じ Agent を異なる粒度で使った場合も、対象スコープと参照文脈を入力情報として渡し、解釈方法は Agent に委ねる。

システムは少なくとも次を区別して管理する。

- 何を分析対象にしたか
- どの粒度の結果か
- 結果をどの一文、段落、章、文書へ紐づけるか
- 本文変更後にどの結果を要再分析として扱うか

マクロ視点では章単位の結果に加え、文書全体の構造を扱う用途がある。文書全体に対する結果は、特定の章へ無理に紐づけず `document` スコープとして保存する。

### 2.5 文脈ポリシー

分析対象そのものは、UI上の視点と実行対象の選択によって決まる。Agent の文脈ポリシーは、対象を評価するためにどこまで参照するかだけを定義する。

Agent ごとに次の文脈ポリシーを選択できる。

| ポリシー | 意味 |
|---|---|
| 先行全文 | 対象より前にある文章をすべて参照する |
| 先行N段落 | 対象より前の直近N段落を参照する |
| 文書全体参照 | 対象より後を含む文書全体を把握しながら、選択された各対象へ個別の結果を返す |

「選択対象だけを分析する」ことは Agent 固有設定ではなく、すべての Agent で利用できる実行UIの責務とする。

文書全体参照は、初見読者の疑似体験とは異なる。どの文脈ポリシーが使われるかをユーザーがAgent設定と実行確認画面で把握できるようにする。

### 2.6 構造化出力

将来の共通出力契約では、回答本文を `response` とする。`deepMeaning` は意味を狭く固定し、自由なAgentの用途と合わないため、新規契約では使用しない。

概念上の最小構造は次の通り。

```json
{
  "response": "Agentの目的に沿った回答",
  "tags": {
    "emotion": ["anxiety"],
    "problem-type": ["duplication"],
    "custom-tag-id": ["custom-value-id"]
  }
}
```

- `response` は必須の文字列
- `tags` と各タグ値は任意
- 該当するタグがなければ、項目の省略または空配列を許容する
- `confidence` は意味が不明確なため新規契約から廃止する
- 回答本文の内容、文体、内部構成は Agent のプロンプトで定義できる
- `response` はMarkdownとして解釈せず、構造化JSON内のテキストとして扱う

既存の `deepMeaning / emotion / theme / confidence` を持つ保存データは、移行時に読み込み互換を維持する。具体的な型変更、保存形式のバージョン更新、移行処理は後続実装で定める。

### 2.7 タグ定義

Agent は、出力に利用するタグ項目を設定できる。

タグ項目には次の2種類がある。

- システムが用意する項目: 感情、問題の種類など
- ユーザーが作成する任意の項目

各タグ項目は、あらかじめ許可された値の集合を持つ。AI はその候補からだけ値を返し、未知の値を生成できないよう、構造化出力のenum等で強制する。

未知の値を受け取った場合も、自動的に新しいタグとして登録・表示・保存しない。表記揺れや無制限な値の増加によって、集計やチャートの意味が失われることを防ぐためである。

タグ値ごとに色を設定できる。色が指定されていない値は、すべて共通のニュートラル色で表示する。システムが自動で未知の意味を推測し、値ごとに異なる色を割り当てない。

将来の感情チャートや文章のボラティリティ可視化では、保存されたタグ値IDを集計単位として使用する。

### 2.8 実行前の透明性

分析実行前の確認画面は既定で表示する。ユーザーは設定で確認画面を非表示にできる。

確認画面では少なくとも次を示す。

- 選択された Reading Agent
- 分析対象の粒度と件数
- 送信される文章量または参照範囲
- 文脈ポリシー
- 追加指示の有無
- 推定使用量

確認画面は、Agentの内容へ介入するためではなく、外部providerまたはローカルLLMへ何が渡るかをユーザーが把握するために存在する。

### 2.9 履歴と来歴

分析履歴は正解や評価の蓄積ではなく、過去に得た観測結果を失わないための記録である。クレジットや外部APIを使って生成した結果が不用意に消えないことを優先する。

履歴UIは主機能として前面に出さず、必要な場合に復旧・参照できる位置へ置く。

結果とともに保存する来歴情報は、結果の意味を最低限判別できる範囲に絞る。

- Agent ID
- 実行時のAgent表示名
- Agentプロンプトのバージョン
- 文脈ポリシー
- 実際に参照した段落数
- 追加指示の有無
- 対象スコープ
- 使用モデル
- 出力契約またはタグ定義を識別するために必要なバージョン

次のテキストは履歴へ複製保存しない。

- Agentプロンプト本文
- 追加指示本文
- 追加指示のハッシュ
- 参照した段落本文
- 文書全体の複製

この方針では過去の分析の完全再現を保証しない。履歴の目的は再現性ではなく、生成済み結果の保持と最低限の識別である。

### 2.10 望ましい出力と避けたいシステム挙動

ここで示す例はAgentの内容を一律に規制するものではなく、ユーザーが設定した目的をシステムが尊重できているかを判断するための例である。

#### 望ましい例

読者Agentが主観的な反応を求められている場合:

> 私は「返事は返ってこなかった」で少し立ち止まった。謝罪が届かないことより、語り手が返事を待っていたことに寂しさを感じた。

編集者Agentが構造上の指摘を求められている場合:

> 結論を示す二文が直前の段落と重複している。この段落を削除しても論旨は失われず、次の具体例への移行が速くなる。

伴走Agentがフリーライティングを支えるよう求められている場合:

> ここでは結論を急がず、今出てきた感覚をそのまま伸ばしてよい。比喩がまだ粗いことより、書き手自身が驚いている箇所に勢いがある。

いずれも、あらかじめ与えられたAgentの目的と評価基準に沿っている。

#### 避けたいシステム挙動

- 読者Agentへシステムが勝手に修正案を要求する
- 編集者Agentへシステムが助言禁止を強制する
- 全Agentへ一般的な「良い文章」の基準を挿入する
- 必要のない感情、テーマ、問題点を必須出力させる
- 意味の不明確な確度を要求する
- AIが返した未知のタグ値を自動登録する
- Agentの目的にかかわらず、事実と推測の区別などの内容規範を一律に追加する
- ユーザー操作なしに分析、再分析、本文変更を行う

### 2.11 現行実装との関係

この文書は分析機能の今後の設計判断における正とする。ただし、2026-06-25時点の実装には次の旧契約が残っている。

- `deepMeaning` を共通回答本文としている
- `emotion / theme / confidence` を全Agentへ要求している

これらは現行データとの互換を保ちながら、後続Issueで段階的に移行する。

- #119: デフォルトReading Agentとプロンプトの見直し

## 3. 制約・非ゴール

- この文書だけで具体的なプロンプト文面を固定しない
- デフォルトReading Agentの一覧を決定しない
- タグ項目・値の全カタログを決定しない
- 既存保存データの移行方式を確定しない
- 章分析、一文分析、文書全体分析のAPI詳細を固定しない
- AIへ本文、ファイル、外部ツールを操作する権限を与えない
- 過去結果の完全な再現性を保証しない

## 4. 検証観点

- Agentのプロンプトが、システムによる不要な内容規範で上書きされない
- 追加指示がAgentプロンプトより優先される
- providerへ構造化出力を要求し、`response` を必須として検証できる
- `confidence` を新規出力契約へ含めない
- 選択されていないタグ項目を要求しない
- 選択されたタグ項目でも空を許容する
- タグ値を許可済み候補へ制限し、未知の値を自動登録しない
- 未指定色のタグ値を共通のニュートラル色で表示する
- 文脈ポリシーをAgent単位で保存・適用できる
- 文書全体を参照しながら対象ごとの結果を返せる
- 分析がユーザーの明示操作なしに実行されない
- 実行確認が既定で表示され、設定により非表示へ変更できる
- 履歴へプロンプト本文、追加指示本文、参照本文を複製保存しない
- 既存の `deepMeaning / emotion / theme / confidence` データを読み込める移行経路を持つ
