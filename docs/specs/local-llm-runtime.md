# Local LLM ランタイム管理方針

関連Issue: #72  
状態: decision（LiteLizard 管理ランタイムは採用しない）

## 結論

LiteLizard は当面、ローカル LLM ランタイムを自前でインストール・起動・停止・更新・削除しない。Local LLM 機能は、ユーザーが別途管理する Ollama 互換 endpoint とモデル名へ接続する方式に限定する。

この判断は、ローカル LLM 対応を捨てるものではない。LiteLizard の責務を「endpoint へ分析リクエストを送ること」に限定し、モデル配布・実行環境・GPU/Metal/CPU 差異・容量管理・プロセス監視を外部ランタイムへ委ねる。

## 比較

| 観点 | 外部ランタイム接続（現行） | LiteLizard 管理ランタイム |
|---|---|---|
| 初期実装 | endpoint URL、モデル名、疎通確認、generate API 呼び出しで済む | モデル一覧、ダウンロード、保存先、起動、停止、更新、削除、異常終了復旧が必要 |
| OS差異 | Ollama 等の外部ランタイムが吸収する | macOS / Windows / Linux、CPU / GPU / Metal の差異をアプリが背負う |
| セキュリティ | 任意 endpoint 送信の説明と入力検証が主 | 外部バイナリ取得、署名、実行、更新、プロセス権限の検証が必要 |
| 容量管理 | ユーザーが外部ランタイム側で管理 | モデル保存先、空き容量、削除、破損復旧をアプリが管理 |
| 障害分離 | ランタイムが落ちても LiteLizard は接続失敗として扱える | 推論プロセスのクラッシュがアプリ体験・サポート負荷に直結する |
| 執筆UIへの影響 | 設定画面の endpoint / model 入力に留まる | モデル管理 UI、進捗、エラー、容量警告が必要になる |
| 保守停止前提との整合 | 高い | 低い |

## 非採用理由

- LiteLizard は文章の読まれ方を観測する執筆環境であり、モデルランタイム配布ツールではない。
- 自前ランタイム管理は、単なる provider 追加ではなく、ネイティブプロセス管理とOS別検証を含む基盤開発になる。
- 未署名 macOS `.dmg` の MVP 配布と組み合わせると、外部バイナリの取得・実行・更新まで扱うことは利用者の警戒とサポート負荷を増やす。
- プロジェクトの今後の人間による継続改善が止まる前提では、モデル配布方式や各OSの推論環境変化を追い続けられない。
- 執筆画面の静けさを優先する設計上、モデル一覧・容量警告・ダウンロード進捗・プロセス状態をアプリ内に持ち込む利益が小さい。

## 現行方式で扱う範囲

LiteLizard が担当するもの:

- Local LLM endpoint URL とモデル名の保存
- endpoint の疎通確認
- 分析実行時の `/api/generate` 呼び出し
- 接続失敗・HTTPエラーの利用者向け表示
- 解析対象本文、参照文脈、Reading Agent prompt、構造化出力 schema の送信

LiteLizard が担当しないもの:

- Ollama 等のインストール
- モデルのダウンロード、保存先管理、削除、更新
- 推論プロセスの起動、停止、アイドル時アンロード
- GPU / Metal / CUDA / CPU の選択
- 外部ランタイムのバージョン管理
- ローカルモデルのライセンス判定

## 将来再検討する条件

次の条件がそろう場合にのみ、別Issueとして再検討する。

- 署名・notarization 済みの配布経路があり、外部バイナリ取得と実行に関する安全説明を整えられる
- 対応OSを限定し、モデル保存先・容量・削除・更新の仕様を保守できる体制がある
- 外部ランタイム接続では解決できない明確なユーザー価値が確認されている
- 執筆UIを騒がせない管理画面分離案がある

この条件を満たさない限り、Local LLM は外部ランタイム接続方式を正とする。
