# Agent Flow（自律エージェント運用ルール）

> **目的**: 複数の自律エージェント（Web Claude Code, Codex 等）と人間が並列に作業しても、衝突・手戻り・レビュー渋滞を最小化するためのラベル運用と Issue 規約を定める。
> **対象読者**: Issue を作成する人/エージェント、Issue を読んで実装する人/エージェント、本リポジトリで稼働するすべての自律ハーネス
> **最終更新**: 2026-05-02

---

## 全体像

```
[Issue 作成]
   │
   │ create-github-issue スキル / 手動
   │ - Depends on: 必要なら明記
   ▼
[Issue が pending 状態]
   │
   │ 人間が判断
   │ - 仕様/設計が固まった → agent-ready ラベル付与
   ▼
[agent-ready Issue]
   │
   │ 自律エージェント（agent-pickup プロンプト）が選択
   │ - 衝突チェック（in-progress Issue 群と内容比較・LLM 判断）
   │ - 同時 in-progress 上限チェック
   │ - in-progress ラベル付与
   ▼
[in-progress Issue]
   │
   │ 実装 → lint/test/build → セルフレビュー → PR 作成
   ▼
[PR レビュー / マージ]
   │
   │ 人間がマージ
   │ - in-progress ラベルを外す
   │ - Issue は close 条件（ユーザー確認等）を満たしてから close
   ▼
[完了]
```

## ラベル定義

### 優先度（既存）
| ラベル | 意味 |
|--------|------|
| `P0` | ブロッカー / 重大破壊 |
| `P1` | MVP・公開に必要 |
| `P2` | 将来改善 / 公開後で良い |

### 種別（既存）
| ラベル | 意味 |
|--------|------|
| `bug` | 期待動作と違う |
| `feature` | 新機能 |
| `docs` | 仕様/WBS/CHANGELOG |
| `refactor` | 振る舞いを変えない整理 |
| `enhancement` | 既存機能の改善（feature と曖昧な場合は feature を優先） |

### 状態（追加分）
| ラベル | 付ける主体 | 意味 |
|--------|------------|------|
| `blocked` | 人間 | 依存タスク未完了・仕様未決・外部要因待ち |
| `agent-ready` | **人間** | 仕様と受け入れ条件が固まり、自律エージェントが拾って実装してよい |
| `in-progress` | **自律エージェント** | エージェントが着手中。PR マージ時に外す |

## Issue 本文の規約

`agent-ready` ラベルを付ける Issue は、`create-github-issue` スキルの標準テンプレ（背景・意図・ゴール・スコープ・非ゴール・受け入れ条件・検証方法）を満たすこと。

### `## Depends on`（必要なら）

```markdown
## Depends on

- #75 (R-18a の型定義が前提)
- #76
```

書き方ルール:
- 依存先 Issue 番号を `#N` 形式で羅列
- 補足が必要なら括弧で1行
- 依存先がすべて close（または PR がマージ済み）になるまで自律エージェントは着手しない

---

## 衝突判定ロジック

自律エージェントが Issue を選ぶときの判定。**ファイル変更範囲を明示するメタデータは持たず、LLM が Issue 本文と既存コードから判断する**。

```
入力:
  candidates = open かつ agent-ready かつ in-progress でない Issue 群
  in_flight  = open かつ in-progress な Issue 群

各 candidate について:
  1. Depends on の依存先がすべて close または PR マージ済みか
     → 否なら skip
  2. candidate と in_flight の各 Issue について、変更範囲が明確に重なるか LLM 判断
     - 同じ機能領域・同じファイル群を触ると判断したら skip
     - 判断に迷ったら安全側に倒して skip
  3. 残った candidate のうち、優先度（P0 > P1 > P2）と古い順で先頭を選ぶ
```

**ポイント**:
- 衝突判定はメタデータでなく LLM の意味的判断に任せる。完璧は求めない
- 判断に迷う場合は skip（人間に拾わせる）
- 依存先が close されていなくても **PR がマージされていれば OK**（Issue close は人間確認待ちの場合があるため）

---

## 同時 in-progress 上限

```
N = 2
```

`in-progress` ラベルが付いている Issue が **2 件以上ある場合、自律エージェントは新規着手しない**。

理由:
- レビューは結局人間 1 人。並列で 3 本以上溜まるとレビュー渋滞 + マージ順の rebase コストが増える
- 上限を超えていたら自律エージェントは「待機」モードで何もしない

人間がローカルで作業する分には上限は気にしなくてよい（エージェントが過剰に並列着手するのを防ぐためのガード）。

## PR 作成ルール

自律エージェントが PR を作るときの規約。

### PR タイトル
- `#<issue番号>: <Issue タイトル相当>` 例: `#55: filesystem IPC のパストラバーサル対策`
- 70 文字以内

### PR 本文の必須項目
- `Closes #<issue番号>`（Issue を自動 close するため）
- 変更概要（3〜5 行）
- 実行した検証（lint/test/build の結果と手動確認）
- セルフレビューで気づいた懸念事項（あれば）

### コミット
- まとめて 1 commit で良い（過度な分割不要）
- メッセージは Issue 番号を冒頭に: `#55 path traversal: validate target path against project root`

## ラベル外し / クローズ運用

| イベント | 操作 |
|----------|------|
| PR 作成 | エージェントは Issue 本文に「PR #N」コメント追加 |
| PR マージ | 人間（または merge hook）が `in-progress` ラベルを外す |
| Issue close | 「ユーザー確認 + CI 通過」が揃ってから人間が close（`read-github-issues` 規約と同じ） |

`in-progress` を外し忘れた場合:
- 次の自律エージェントが衝突判定で誤って skip するリスクがある
- 24 時間以上 PR が動いていない `in-progress` Issue は人間がレビューして外す

## よくある質問

**Q. 小さな typo 修正に Issue / agent-ready は必要？**
A. 不要。自律エージェント運用は「ある程度のスコープがある実装」用。typo は手元で直接修正してコミット。

**Q. 人間が手元で作業する Issue にも `in-progress` を付ける？**
A. 推奨。自律エージェントの衝突判定対象になり、二重作業を防げる。

**Q. `agent-ready` を付ける基準は？**
A. 以下を満たすこと：
  - 受け入れ条件が観測可能
  - 依存先（あれば `Depends on:`）が明記されている
  - 仕様/設計が固まっており、エージェントが「いい感じに」判断する余地が少ない

**Q. 人間と自律エージェントで `agent-ready` の扱いはどう違う？**
A. 視点で逆転する：
  - **自律エージェント**: `agent-ready` 付きの中からしか拾わない（必須条件）
  - **人間**: `agent-ready` 付きは自律エージェントに任せられるため、自分の時間は **`agent-ready` が付いていない Issue**（仕様検討・設計判断・複雑な調整が必要なもの）に充てる
  - 結果として、人間は agent-ready 化前のタスクを、エージェントは agent-ready 化後のタスクを処理する分業になる
