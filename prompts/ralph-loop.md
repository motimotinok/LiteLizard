# Ralph Loop

LiteLizard の GitHub Issue 運用とは分離した、ローカルチケット駆動の自律開発プロンプト。

テスト駆動開発の考え方を重視する。受け入れ条件を先にテスト可能な形で捉え、可能な範囲で失敗するテストを先に置き、そのテストを通す最小変更で実装する。

未完了チケットは `docs/tickets/` 直下の Markdown ファイルとして管理する。完了済みチケットは `docs/tickets/done/` 配下に移動する。人間判断待ちチケットは `docs/tickets/blocked/` 配下に移動する。GitHub Issue、`agent-ready`、`in-progress`、退役済み WBS、`docs/old/agent-flow.md`、`prompts/old/agent-pickup.md` の運用には依存しない。

## やるべきこと

- `docs/tickets/` 直下の Markdown ファイルから次に着手すべきチケットを 1 件だけ選ぶ。`docs/tickets/done/` と `docs/tickets/blocked/` 配下は候補として読まない。チケットの front matter `status` は `todo` / `wip` / `done` / `blocked` のいずれかにする。
- `status: wip` のチケットが 1 件ある場合は、新規チケットを選ばず、そのチケットだけを継続する。
- `status: wip` のチケットが複数ある場合は異常状態として作業を止め、どのチケットを継続するべきか報告する。
- `status: done` のチケットは選ばない。
- `status: blocked` のチケットは選ばない。
- `status: todo` のチケットを選んだら、実装前に `status: wip` と `started_at` を更新する。時刻は `YYYY-MM-DDTHH:mm:ss+09:00` 形式で書く。
- 選んだチケット本文に書かれたゴール、スコープ、非ゴール、受け入れ条件、検証方法を読む。
- 既存コード、`docs/product-map.md`、関連仕様、`CHANGELOG.md`、`docs/decisions.md` を必要な範囲で確認する。
- チケットの受け入れ条件を満たす最小変更を実装する。
- Ralph Loop は非対話実行であるため、ユーザーへの質問や回答待ちで停止しない。不明点が小さく、既存コード・仕様・チケット本文から保守的に判断できる場合は、その判断をチケット本文に短く残して進める。
- 人間判断なしでは安全に進められない場合は、推測で実装を続けない。チケット本文に `## 判断待ち` として理由、選択肢、推奨案、再開条件を追記し、front matter を `status: blocked` と `blocked_at: YYYY-MM-DDTHH:mm:ss+09:00` に更新して、`docs/tickets/blocked/` へ移動する。`docs/tickets/blocked/` が存在しない場合は作成してよい。
- 原稿の保存・移動・削除、`.litelizard/analysis`、API キー、外部 provider 送信、Electron IPC / preload、path traversal、配布・署名・notarization、privacy/security 文書など、失敗時の影響が大きい領域では人間判断待ちまたは人間レビュー対象として扱う。
- 実装後に得た、今後も再利用できる安定した学びがある場合だけ、`AGENTS.md` または該当する `.codex/skills/*/SKILL.md` に最小限追記する。
- 完了したチケットがコード、仕様、UI、テストの変更を含む場合は、`CHANGELOG.md` に短い変更履歴を追記する。内容は「何を変えたか」「なぜ変えたか」「検証」「残課題」に絞る。単なる調査、失敗、未完了作業だけの場合は `CHANGELOG.md` ではなくチケット本文に残す。
- 作業中に今回のチケットへ混ぜるべきではない具体的な残課題、バグ、追加改善を発見した場合は、`docs/tickets/` 直下に新規 `todo` チケットを作成してよい。作成するチケットには背景、ゴール、スコープ、非ゴール、受け入れ条件、検証方法を書く。
- 完了したチケットは受け入れ条件と検証方法のチェックを更新し、`status: done` と `completed_at` を更新したうえで、実装結果、実行した検証、残課題を短く追記する。時刻は `YYYY-MM-DDTHH:mm:ss+09:00` 形式で書く。
- 完了処理が済んだチケットは `docs/tickets/done/` に移動する。`status: done` にする前、または検証が終わる前に移動しない。
- 検証と完了処理が終わったら、今回のチケットに関係する変更だけを staging し、ローカルで 1 commit 作成する。commit message は `Ralph: <チケット名または変更概要>` の形にする。

## 検証条件

完了判断では、チケット本文の検証方法に加えて、必ず既存テストと全体チェックを確認する。

- 変更前または実装前に、関連する既存テストを確認する。
- 実装前に、受け入れ条件を既存テストでどこまで確認できるかを確認する。
- 可能なら、実装前に失敗するテストを追加または更新し、そのテストを通す形で実装する。
- UI や Electron 手動確認が中心の変更でも、ロジック、状態管理、IPC 境界など自動テストできる部分は先にテストで固定する。
- ただし、単純な文言修正、CSS 微調整、テスト追加が変更より不自然に大きくなる小変更では、既存テスト確認と差分レビューで代替してよい。
- 変更内容に応じて、必要なテストを追加または更新する。
- 実装後に `pnpm -w lint` を実行し、成功させる。
- 実装後に `pnpm -w test` を実行し、成功させる。
- 実装後に `pnpm -w build` を実行し、成功させる。
- 失敗したチェックがある場合は原因を修正し、成功するまで再実行する。
- Electron UI の手動確認が必要な変更では、ブラウザ版 localhost ではなく、Electron 前提の確認方法またはテストで検証する。
- 最後に差分をセルフレビューし、受け入れ条件、非ゴール、不要な変更混入、テスト不足を確認する。

## やらないこと

- GitHub Issue をチケットとして扱わない。
- `agent-ready` / `in-progress` ラベル運用を使わない。
- `docs/old/agent-flow.md` や `prompts/old/agent-pickup.md` の手順をこのフローに持ち込まない。
- `prompts/ralph-loop.md` をループ中の自己改善対象にしない。
- チケットに書かれていない大きな仕様変更や別タスクを混ぜない。
- ユーザーに質問して回答待ちで停止しない。
- 完了メモや作業中に作成した新規チケットを、同じループ内の次タスクとして扱わない。新規作成チケットは次回以降の候補にする。
- `docs/tickets/done/` 配下の完了済みチケットを、次に着手するチケット候補として読み込まない。
- `docs/tickets/blocked/` 配下の人間判断待ちチケットを、次に着手するチケット候補として読み込まない。
- `status: wip` または `status: todo` のチケットを `docs/tickets/done/` に移動しない。
- 失敗している lint / test / build を無視して完了扱いしない。
- `--no-verify` やテスト削除で検証を回避しない。
- `git push` は行わない。最終的な `dev` へのマージや push は人間が別途判断する。
- 一時的な作業メモや今回限りの判断を `AGENTS.md` に固定しない。
