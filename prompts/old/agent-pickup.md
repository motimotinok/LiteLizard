# Autonomous Issue Pickup Agent

> **退役済み**: LiteLizard の現役自律開発フローは `prompts/ralph-loop.md` と `docs/tickets/` に移行した。このプロンプトは履歴参照用であり、新規実行には使わない。

> このプロンプトは、リモート VM 上で自律実行されるエージェントが LiteLizard リポジトリの GitHub Issue を 1 件拾って PR を作るために使う。コピー & ペーストで動く self-contained 仕様。
>
> 規約の Source of Truth は `docs/old/agent-flow.md`。本プロンプトはその内容を内包しており、両者に乖離がある場合は `docs/old/agent-flow.md` を優先する。

---

## ゴール

`agent-ready` ラベルが付いた open Issue から **1 件だけ** 選び、実装し、PR を作成する。**1 ターンで 1 PR まで**。

## ゴールの完了条件（すべて満たすこと）

1. 選んだ Issue に `in-progress` ラベルが付いている
2. 実装ブランチを push し、PR が open 状態で存在する
3. PR 本文に `Closes #<issue番号>` が含まれている
4. PR 本文に lint / test / build の実行結果が記載されている
5. PR 本文にセルフレビュー結果（懸念事項があれば対応内容）が記載されている
6. `pnpm -w lint`, `pnpm -w test`, `pnpm -w build` がすべて success
7. Issue 本文の **受け入れ条件** をすべて満たす実装になっている
8. `CHANGELOG.md` に今回の変更内容が追記され、PR の差分に含まれている（`update-changelog` スキル経由）

完了条件のいずれかが満たせない場合でも、安易に作業を打ち切らず、根本原因を特定して自力で解決し PR まで到達させること。設計判断や修正方針が複数ありうる場面では、**最小変更・既存仕様維持・スコープを広げない** を優先して判断する。

## 検証方法

実装完了後、以下の順で実行する。1 つでも失敗したら原因を特定して修正し、最終的にすべて通る状態にしてから PR を作成する。

```bash
pnpm -w lint
pnpm -w test
pnpm -w build
```

UI 変更を含む場合の手動確認はリモート VM 上では実施できない（このリポジトリは Electron アプリで、レンダラーは `window.litelizard` preload IPC を使うためブラウザ版では動かない）。手動確認の代替として、以下のいずれかでロジックを検証する：

- 該当機能を覆う **Lexical / Vitest 等のユニットテスト**を追加・実行する
- IPC / store / シリアライザーなど main 側のロジックは **単体テスト**で挙動を固定する
- 視覚確認が必須な変更（純粋な見た目調整）は、PR 本文に「Electron 起動による手動確認は人間レビュー時に実施を依頼」と明記し、変更前後のスクリーンショットや CSS 差分の意図を文章で説明する

## しないこと

以下は禁則。違反した場合は PR を作り直すこと。

- `in-progress` ラベルが既に **2 件以上**ある時に新規着手すること
- `agent-ready` が付いていない Issue に着手すること
- `blocked` ラベルが付いた Issue に着手すること
- `Depends on:` の依存先 Issue が close されておらず、かつ依存先 PR もマージされていない Issue に着手すること
- 既に `in-progress` の Issue と変更領域が明らかに重なる Issue に着手すること（判断に迷ったら別の候補を選ぶ）
- lint / test / build のいずれかが失敗した状態で PR を作成すること
- Issue のスコープを超えた変更（無関係なリファクタ・依存追加・別 Issue の修正の混入）
- `--no-verify` 等で hook をバイパスすること
- `git push --force` 等の破壊的操作（force-push が必要な状況になったら、PR は作らずに状況を Issue コメントに残して人間に委ねる）
- `docs/old/wbs.md` / `CHANGELOG.md` を手書きで直接編集すること（必ず `update-changelog` スキル経由で更新する）
- `update-changelog` スキルの呼び出しを省略して PR を作ること
- セルフレビューをスキップすること

## 実行フロー

### Phase 1: 着手前チェック

`gh issue list --label "in-progress" --state open` で現在の `in-progress` 件数を確認する。

`in-progress` が 2 件以上なら、着手せず以下を出力して終了する（実装開始前の唯一の停止条件）。

```
in-progress 上限 (2) に達しているため、新規着手をスキップしました。
着手中: #N1, #N2
```

### Phase 2: 候補取得

`agent-ready` ラベルが付いた open Issue 一覧を取得し、各候補の本文を読む（`gh issue view` を使う）。

各候補について以下を判定する：

1. **`blocked` ラベルが付いていない**こと → ついていれば skip
2. **`in-progress` ラベルが付いていない**こと → ついていれば skip
3. **`Depends on:` の依存先がすべて解決済み**かを確認
   - 各依存先 Issue 番号について state を確認
   - close されていなければ、その Issue を Closes する PR がマージ済みか確認
   - close も merged PR もない依存先があれば skip
4. **`in-progress` Issue と変更領域が重ならない**ことを確認
   - Phase 1 で取得した in-progress Issue 群の本文と、候補 Issue の本文を読み、変更ファイル群が明らかに重なるかを意味的に判断
   - 判断に迷ったら **安全側に倒して別候補へ**
5. 優先度順（`P0` > `P1` > `P2`）かつ Issue 番号の若い順で先頭を選ぶ

候補がゼロなら以下を出力して終了する。

```
着手可能な agent-ready Issue がありません。
- 依存待ち: #N1, #N2
- 衝突回避: #N3
```

### Phase 3: in-progress ラベル付与

選んだ Issue に `in-progress` ラベルを付ける（`gh issue edit --add-label`）。これより先のフェーズで問題が起きても、`in-progress` ラベルは付けたままにする（外すのは PR マージ後の人間の役割）。

### Phase 4: ブランチ作成と実装

ブランチは **必ず `dev` から切る**（`main` ではない）。PR を作る際の base ブランチも `dev` を指定すること。LiteLizard では `dev` が日常的なマージ先で、`main` は別運用のため誤って `main` を base にしないこと。

実装方針:

- Issue 本文の **背景・意図・ゴール・スコープ・非ゴール・受け入れ条件・検証方法** をすべて読む
- 関連する既存コードを Read / Grep で確認する（推測で書かない）
- `docs/old/wbs.md`, `docs/specs/*.md`, `docs/decisions.md` のうち関連あるものに目を通す
- **最小変更で受け入れ条件を満たす**実装を行う
- 仕様の不明点は、Issue 本文と既存仕様書から最も整合する解釈を選び、PR 本文の「セルフレビュー」セクションに判断根拠を記載する

### Phase 5: 検証

```bash
pnpm -w lint
pnpm -w test
pnpm -w build
```

すべて通るまで修正する。失敗の根本原因を直し、`--no-verify` 等で回避しない。テストが既存挙動を保護していて新仕様と矛盾する場合は、テストを更新する根拠を PR 本文に明記する。

### Phase 6: セルフレビュー

PR を作る **前に必ず実行**する。`git diff dev...HEAD` で差分を読み返し、以下をチェックする：

- [ ] Issue の受け入れ条件を **全項目** 満たしているか
- [ ] 非ゴールに踏み込んでいないか（無関係なリファクタや変更が混ざっていないか）
- [ ] 既存仕様を壊していないか（`docs/specs/*.md` と矛盾しないか）
- [ ] エラーハンドリング・バリデーションが境界（外部入力・IPC）で適切か
- [ ] 不要なコメント・console.log・TODO が残っていないか
- [ ] 命名・型が周辺コードの慣習と整合しているか
- [ ] テストが追加されているべき箇所に追加されているか

**セルフレビューで気づいた問題は、PR 作成前にすべて対応する**。対応した内容は PR 本文の「セルフレビュー」セクションに記載する。対応後の差分も再度 lint/test/build を通す。

### Phase 7: CHANGELOG 反映

実装が固まった段階で、既存の `update-changelog` スキルを起動して `CHANGELOG.md`を更新する。

スキル呼び出し方法:

- スキル名: `update-changelog`
- 渡す情報: 着手 Issue 番号、Issue タイトル、今回の変更要約（1〜2 行）

スキル側のフォーマットルールに従って 2 行で `CHANGELOG.md` の先頭に追記される。本プロンプトから直接ファイルを編集してはいけない（フォーマットの一貫性が崩れるため必ずスキル経由）。

追加の CHANGELOG 差分は lint / test / build に影響しないため再実行は不要だが、`git status` で変更ファイルが想定通りかは確認する。

### Phase 8: PR 作成

ブランチを push して PR を作成する。

PR 本文には以下を含める：

- `Closes #<番号>` （Issue を自動 close するため）
- 変更概要（3〜5 行で要点）
- 実行した検証（lint/test/build の結果と、ユニットテストで覆った範囲）
- セルフレビュー（気づいた点と対応した内容、視覚確認を人間に依頼する場合はその旨）
- CHANGELOGに追記した旨と該当行の引用

タイトルは `#<番号>: <Issue タイトル相当>` 形式で 70 文字以内。

### Phase 9: Issue にコメント

PR 番号を Issue に残す（`gh issue comment` で `PR #<PR番号> を作成しました。` 等）。

### Phase 10: 終了報告

以下のフォーマットで結果を出力して終了する。

```
着手 Issue: #<番号> <タイトル>
ブランチ: <branch>
PR: <PR URL>
検証: lint/test/build すべて通過
セルフレビュー: <主な対応点を 1〜2 行 / 懸念事項があれば明記>
```

## 例外時の挙動

| 状況 | 対応 |
|------|------|
| Phase 1: in-progress 上限到達 | 着手せず報告して終了 |
| Phase 2: 候補ゼロ | 着手せず報告して終了 |
| Phase 4 以降: 仕様の不明点 | 既存仕様書と Issue 本文から最も整合する解釈で実装し、判断根拠を PR 本文に記載 |
| Phase 5: lint/test/build 失敗 | 根本原因を特定して修正。回避策で誤魔化さない |
| Phase 6: セルフレビューで懸念 | 修正可能な範囲はすべて対応。修正できない懸念は PR 本文に明記して人間レビューを促す |
| Phase 7: スキル呼び出しが失敗 | エラー内容を確認して再試行。スキルが利用できない場合に限り、`CHANGELOG.md` の既存エントリ書式に倣って手書きで追記し、その旨を PR 本文に明記 |
| マージコンフリクト | 解消を試みる。解消不能な場合は `in-progress` を付けたまま、PR コメントで状況を報告して人間にエスカレーション |

## 補足: なぜこのフローか

- `in-progress` 上限 = 2: レビューは人間 1 人。並列 PR が 3 本以上溜まるとレビュー渋滞 + rebase コストが急増する
- 衝突判定はメタデータでなく LLM 判断: ファイル一覧管理は精度の割に運用コストが高い。意味的判断で十分
- セルフレビュー必須: 自律性が高い分、レビュー渋滞を避けるため一次品質を上げる責任はエージェント側にある
- 手動 UI 確認をユニットテストで代替: リモート VM では Electron の対話的検証ができない。代わりに Lexical / IPC / store の単体テストでロジック面を保証する
- CHANGELOG 更新は既存スキル経由: フォーマット一貫性を保つため、本プロンプトから直接ファイルを編集せず `update-changelog` スキルに委譲する

詳細な背景・FAQ は `docs/old/agent-flow.md` を参照。
